# Plan: T-2026-371 — Create integration test for SettingsService theme preference applying body class

## Task

Create an integration test that verifies the full chain: change theme via SettingsService -> body class updates -> old class removed. Uses real SettingsService with document body access (not mocked DOCUMENT).

## Architecture

This is a test-only ticket. No production code changes. The integration test exercises:
1. SettingsService (real, `providedIn: 'root'`)
2. StatePersistenceService (real, backed by fake localStorage)
3. `document.body` (real DOM body element)

The test verifies that theme changes flow through the signal/effect chain and manifest as CSS classes on the body element.

**Note on acceptance criteria:** The ticket says "default theme -> body has `theme-dark` class" but the actual default in `getDefaultSettings()` is `theme: 'station'`. The test will verify the actual default (`theme-station`), matching the implementation.

## Test Plan

Four test cases matching the acceptance criteria:

1. **Default theme applies `theme-station` class** — Inject SettingsService with no persisted settings, verify body has `theme-station`.
2. **Change to station theme** — Update theme to `'dark'` first, then to `'station'`, verify body has `theme-station` and not `theme-dark`.
3. **Change to light theme** — Update theme to `'light'`, verify body has `theme-light` and old class removed.
4. **Theme persists after simulated reload** — Save theme `'light'` to fake storage, create a fresh TestBed/SettingsService, verify it loads `theme-light` and applies the class.

Plus additional coverage:
5. **Switching through all three themes** — Cycle through all themes, verify only one theme class at a time.
