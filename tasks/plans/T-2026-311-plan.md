# Plan: T-2026-311 â€” Create integration test for MinigameEngine auto-pause on page visibility change

## Task

Create an integration test file at `src/app/core/minigame/auto-pause.integration.spec.ts` that verifies the MinigameEngine auto-pause behavior when the browser tab loses/gains visibility. Uses a real MinigameEngine subclass (not mocks) and simulates `visibilitychange` events on `document`. Must verify: auto-pause on tab hide, auto-resume on tab show, manual pause precedence over auto-resume, onPause/onResume lifecycle hooks, and timer elapsed time excluding paused duration.

## Architecture

This is a single integration test file with no production code changes. The key design decisions:

1. **Real subclass, not mocks.** The ticket explicitly requires a real MinigameEngine subclass for true integration testing. We create a `TestableEngine` class that extends `MinigameEngine<{ difficulty: number }>` with minimal implementations of the abstract methods and tracking flags for `onPause`/`onResume` calls. This mirrors the `TestEngine` pattern already used in `minigame-engine.spec.ts`. The `TestableEngine` is defined locally in this file (not shared) because it has integration-specific tracking flags (`onPauseCalled`, `onResumeCalled`) that the unit test's `TestEngine` does not need. Sharing a helper would couple the two test files unnecessarily.

2. **Real document, not mock document â€” why this adds value.** The existing unit tests in `minigame-engine.spec.ts` use `createMockDocument()` which provides a simplified `addEventListener`/`removeEventListener`/`dispatchEvent` implementation. The mock cannot catch bugs related to: (a) JSDOM/browser-specific `document.hidden` getter behavior, (b) real `EventTarget` dispatch ordering, (c) engine constructor's `typeof document !== 'undefined'` detection path, or (d) interaction between multiple event listeners on the real document. This integration test exercises the real `document` API path that end users actually hit, providing confidence that the mock's contract matches reality.

3. **Vitest fake timers for timer verification.** We use `vi.useFakeTimers()` and `vi.advanceTimersByTime()` to verify that timer elapsed time excludes auto-paused duration. Fake timers are scoped to a nested `describe` block with `beforeEach`/`afterEach` to prevent leaking into other tests if a test throws.

4. **Why a separate file instead of adding to the existing spec.** The existing `minigame-engine.spec.ts` already has unit-level visibility tests using mock documents. This integration spec tests the same feature through the real `document` API. The `.integration.spec.ts` naming convention is already established in the project (see `core-progression.integration.spec.ts`, `minigame-play.integration.spec.ts`).

## Test Plan

The file contains exactly 5 test cases matching the acceptance criteria:

### Test 1: Auto-pause on tab hide (verifies `_autoPaused` behaviorally)
- Create a `TestableEngine` with no `document` override (uses real `document`)
- Initialize and start the engine (status = Playing)
- `simulateVisibilityChange(true)` â€” set `document.hidden = true` + dispatch event
- Assert: `engine.status()` is `MinigameStatus.Paused`
- **Behavioral `_autoPaused` verification:** immediately call `simulateVisibilityChange(false)` and assert status returns to `Playing`. This confirms `_autoPaused` was set (only auto-paused engines auto-resume). Each `it` block gets a fresh engine from `beforeEach`, so no state restoration is needed.

### Test 2: Auto-resume on tab show
- Start engine, auto-pause via `simulateVisibilityChange(true)`
- `simulateVisibilityChange(false)` â€” set `document.hidden = false` + dispatch event
- Assert: `engine.status()` is `MinigameStatus.Playing`

### Test 3: Manual pause takes precedence over auto-resume
- Start engine (initialize + start â€” status = Playing)
- Call `engine.pause()` manually â€” status becomes Paused
- `simulateVisibilityChange(true)` â€” dispatched but is a no-op because engine is already Paused and the handler requires `Playing` status for auto-pause. This step is included explicitly to verify the handler does not double-set `_autoPaused` on an already-paused engine.
- `simulateVisibilityChange(false)` â€” handler checks `_autoPaused` which is `false` (manual pause), so does NOT auto-resume
- Assert: `engine.status()` remains `MinigameStatus.Paused` â€” manual pause is preserved

### Test 4: onPause and onResume lifecycle hooks fire during auto-pause/resume
- Start engine (fresh instance, tracking flags are `false`)
- `simulateVisibilityChange(true)` â€” auto-pause
- Assert: `engine.onPauseCalled` is `true`
- `simulateVisibilityChange(false)` â€” auto-resume
- Assert: `engine.onResumeCalled` is `true`

### Test 5: Timer elapsed time excludes auto-paused duration (in nested `describe` with fake timers)
- Create engine with `timerDuration: 30` (no `document` override)
- Use `vi.useFakeTimers()` scoped to a nested `describe` block with `beforeEach`/`afterEach`
- Initialize, start, advance 5s (timeRemaining = 25)
- `simulateVisibilityChange(true)` â€” auto-pause
- Advance 10s while paused (timer should be frozen)
- Assert: `timeRemaining` is still 25
- `simulateVisibilityChange(false)` â€” auto-resume
- Advance 3s (timer resumes)
- Assert: `timeRemaining` is 22 (only 5+3=8s of actual play elapsed out of 30)
- `vi.useRealTimers()` in nested `afterEach` (guaranteed cleanup even if test throws)

## Detailed Implementation

### TestableEngine subclass

```typescript
class TestableEngine extends MinigameEngine<{ difficulty: number }> {
  onPauseCalled = false;
  onResumeCalled = false;

  constructor(config?: Partial<MinigameEngineConfig>) {
    super(config);
  }

  protected onLevelLoad(): void { /* no-op */ }
  protected onStart(): void { /* no-op */ }
  protected onComplete(): void { /* no-op */ }
  protected validateAction(): ActionResult {
    return { valid: true, scoreChange: 10, livesChange: 0 };
  }
  protected override onPause(): void { this.onPauseCalled = true; }
  protected override onResume(): void { this.onResumeCalled = true; }
}
```

### simulateVisibilityChange helper

```typescript
function simulateVisibilityChange(hidden: boolean): void {
  Object.defineProperty(document, 'hidden', {
    configurable: true,
    get: () => hidden,
  });
  document.dispatchEvent(new Event('visibilitychange'));
}
```

### beforeEach / afterEach (outer describe)

```typescript
let engine: TestableEngine;
let savedDescriptor: PropertyDescriptor | undefined;

beforeEach(() => {
  savedDescriptor =
    Object.getOwnPropertyDescriptor(document, 'hidden') ??
    Object.getOwnPropertyDescriptor(Object.getPrototypeOf(document), 'hidden');
  engine = new TestableEngine();
  engine.initialize(createTestLevel());
  engine.start();
});

afterEach(() => {
  engine.destroy();
  // Delete the instance override on document.hidden â€” falls back to the prototype getter.
  // In JSDOM, `hidden` is always a prototype getter, so delete is sufficient.
  if (Object.getOwnPropertyDescriptor(document, 'hidden')) {
    delete (document as any)['hidden'];
  }
});
```
