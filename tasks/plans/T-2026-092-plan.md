# Plan: T-2026-092 — Create Power Grid minigame engine

## Task

Implement the Power Grid engine: service-component wiring, injection scope validation, and circuit board puzzle logic. The engine extends `MinigameEngine<PowerGridLevelData>` and supports actions for connecting/disconnecting services to components with scope selection, validating connections against the answer key, and detecting short circuits (wrong connections).

## Architecture

### Design Decisions

**1. Engine-only inline validation (no external service yet).** T-2026-433 will later create `PowerGridInjectionService` as an optional constructor-injected service (matching the Wire Protocol and Flow Commander patterns). This engine must work standalone with inline validation logic, and expose a constructor slot for the future service. The inline validation compares player connections against `validConnections` (the static answer key) and `scopeRules`.

**2. Runtime `PowerConnection` type defined in the engine file.** T-2026-270 (data model ticket) is still `todo` and will add `PowerConnection`, `ShortCircuit`, and other runtime types to `power-grid.types.ts`. Since T-2026-270 is not a dependency of this ticket, the engine defines its own local types for player connections and short circuits. These types will be reconciled when T-2026-270 is implemented.

**3. Two actions + standalone verify() method.** Following Wire Protocol's exact pattern:
- `connect-service`: Wire a service to a component with a chosen scope. Validates: service exists, component exists, component requires that service, no duplicate (same serviceId+componentId). Does NOT validate scope correctness at connect time — players experiment freely. Scope is only checked during `verify()`.
- `disconnect-service`: Remove an existing connection by ID.
- `verify()`: Standalone method (NOT an action through submitAction()). Compares all connections against the answer key. Wrong connections = short circuits. All correct = win. Verification attempts limited to `DEFAULT_MAX_VERIFICATIONS` (3, hard-coded — `PowerGridLevelData` has no `maxVerifications` field).

**4. Short circuit = any connection that is wrong.** Two reasons for short circuits:
- `'wrong-pair'`: service wired to a component that doesn't need it (no matching ValidConnection exists)
- `'wrong-scope'`: correct service-component pair but incorrect injection scope

Missing connections (component has unmet injections) are tracked separately in `GridValidationResult.missingConnections` as `ValidConnection[]` entries — they are NOT ShortCircuitInfo entries. Duplicate connections cannot exist because `handleConnect` rejects them at connect-time.

**5. Scoring follows Wire Protocol's attempt-based model.** First-attempt perfect = 100%, degrading with each failed verification. This fits the "activate and verify" gameplay described in the spec.

**6. Verification service interface.** Define a `PowerGridInjectionService` interface (matching the class name T-2026-433 will implement) that the engine can optionally delegate to. Only `validateAll()` is used by the engine (in `verify()`). `validateConnection()` is intentionally omitted — scope validation only happens at verify time, not at connect time.

### Data Flow

```
Level data (PowerGridLevelData) -> onLevelLoad() -> populates signals
Player actions (connect/disconnect) -> submitAction() -> validateAction() -> updates _connections signal
Validate -> verify() -> compare _connections against validConnections -> ShortCircuitResult
All correct -> addScore() + complete()
3 failed verifications -> fail()
```

## Detailed Implementation

### `power-grid.engine.ts` Structure

#### Local types (defined in engine file, not in types file)

```typescript
/** A player-drawn connection between a service and component. */
export interface PowerConnection {
  readonly id: string;
  readonly serviceId: string;
  readonly componentId: string;
  readonly scope: InjectionScope;
}

/** A short circuit detected during validation. */
export interface ShortCircuitInfo {
  readonly connectionId: string;
  readonly serviceId: string;
  readonly componentId: string;
  readonly reason: 'wrong-pair' | 'wrong-scope';
}

/** Result of validating all connections. */
export interface GridValidationResult {
  readonly correctConnections: readonly PowerConnection[];
  readonly shortCircuits: readonly ShortCircuitInfo[];
  readonly missingConnections: readonly ValidConnection[];
  readonly allCorrect: boolean;
}
```

#### Action types

```typescript
export interface ConnectServiceAction {
  readonly type: 'connect-service';
  readonly serviceId: string;
  readonly componentId: string;
  readonly scope: InjectionScope;
}

export interface DisconnectServiceAction {
  readonly type: 'disconnect-service';
  readonly connectionId: string;
}

export type PowerGridAction = ConnectServiceAction | DisconnectServiceAction;
// Note: verify() is a standalone method, NOT an action type.
```

#### Validation service interface

```typescript
export interface PowerGridInjectionService {
  validateAll(
    connections: readonly PowerConnection[],
    services: readonly ServiceNode[],
    components: readonly ComponentNode[],
    validConnections: readonly ValidConnection[],
    scopeRules: readonly ScopeRule[],
  ): GridValidationResult;
  reset?(): void;
}
```

#### Engine class signals

Private writable signals:
- `_connections: signal<readonly PowerConnection[]>([])` — player's current connections
- `_validationResult: signal<GridValidationResult | null>(null)` — last verification result (exposed as public `validationResult` signal for UI to display last-result feedback)
- `_services: signal<readonly ServiceNode[]>([])` — services from level data
- `_components: signal<readonly ComponentNode[]>([])` — components from level data
- `_validConnections: readonly ValidConnection[]` — answer key (private field, not signal)
- `_scopeRules: readonly ScopeRule[]` — scope rules (private field, not signal)
- `_verificationsRemaining: signal<number>(3)` — max verification attempts
- `_verificationCount: signal<number>(0)` — how many verifications used
- `_nextConnectionId: number` — auto-incrementing ID counter

Public readonly signals expose all of the above (except answer key).

#### Key methods

- `onLevelLoad(data)`: populate signals from level data, reset connections
- `onStart()`: empty (no timer needed for this puzzle)
- `onComplete()`: empty
- `validateAction(action)`: dispatch to `handleConnect` or `handleDisconnect`
- `handleConnect(action)`: validate serviceId exists, componentId exists, component requires service, no duplicate (same service+component), then add connection
- `handleDisconnect(action)`: validate connectionId exists, then remove
- `verify(): GridValidationResult | null`: compare all connections against answer key. Correct = matching (serviceId, componentId, scope) tuple. Short circuit = wrong pair or wrong scope. Missing = required connections not present. If all correct, addScore + complete. If verificationsRemaining hits 0, fail.
- `calculateScore()`: attempt-based scoring matching Wire Protocol pattern

#### Inline validation logic

The `verify()` method performs inline validation (no service needed):

1. For each player connection, check if a matching `ValidConnection` exists with same (serviceId, componentId, scope).
2. If serviceId+componentId match but scope differs -> `wrong-scope` short circuit.
3. If no match at all -> `wrong-pair` short circuit.
4. Find missing connections: `validConnections` entries with no matching player connection. These go in `GridValidationResult.missingConnections` (NOT as ShortCircuitInfo).
5. `allCorrect` = no short circuits AND no missing connections.

Note: Duplicate connections cannot appear because `handleConnect` rejects them. No duplicate check in verify().

When a `PowerGridValidationService` is provided, delegate to `validateAll()` instead.
