# Plan: T-2026-057 — Create ConfirmDialogComponent for destructive actions

## Task

The settings page specifies "Reset progress (with confirmation)" and the minigame shell has a "Quit" action in the pause menu that should confirm before abandoning a level. This shared dialog component handles destructive action confirmations.

Acceptance criteria:
- `ConfirmDialogComponent` at `src/app/shared/components/confirm-dialog/`
- Inputs: `title` (string), `message` (string), `confirmLabel` (string, default "Confirm"), `cancelLabel` (string, default "Cancel"), `variant` ('danger' | 'warning')
- Outputs: `confirmed` event, `cancelled` event
- Modal overlay with semi-transparent backdrop
- Danger variant uses Emergency Red for confirm button
- Warning variant uses Alert Orange for confirm button
- Keyboard accessible: Escape to cancel, Enter to confirm, focus trap within dialog
- Unit tests for: rendering, confirm/cancel event emission, keyboard handling, focus trap

## Architecture

### Key Design Decision: Native `<dialog>` Element

The component will use the native HTML `<dialog>` element with its `.showModal()` API instead of a custom overlay `<div>`. Rationale:

1. **Built-in accessibility.** The `<dialog>` element opened via `.showModal()` automatically provides `role="dialog"`, `aria-modal="true"`, focus trapping, and Escape-to-close behavior. This satisfies three acceptance criteria (modal overlay, focus trap, Escape key) without custom code.
2. **Native backdrop.** The `::backdrop` pseudo-element provides the semi-transparent overlay, eliminating the need for a custom backdrop `<div>`.
3. **No CDK dependency.** The project does not use `@angular/cdk` and adding it for a single focus-trap utility would be excessive.

### Component API

The component is designed to be placed in a parent template and programmatically shown/hidden via a boolean signal or `@if` block. It uses `afterNextRender` to call `showModal()` after the `<dialog>` element is inserted into the DOM.

**Why `afterNextRender` + `showModal()` instead of the `open` attribute?** The native `open` attribute on `<dialog>` does NOT show it as a modal — it shows it as a non-modal dialog without backdrop or focus trap. Only `.showModal()` provides the full modal behavior we need.

**Note on `variant` default:** The AC lists `variant` as `('danger' | 'warning')` without specifying a default. We default to `'danger'` since the primary use case (reset progress, quit level) is destructive. This is a design choice, not an AC violation.

### Open/Close Pattern

The ConfirmDialogComponent will call `showModal()` in `afterNextRender` so that whenever the component is rendered (e.g., via `@if (showDialog())` in the parent), the dialog immediately opens as a modal. When the user confirms or cancels, the component emits the appropriate event and calls `dialog.close()`. The parent controls visibility by toggling the `@if` condition.

### Keyboard Handling

- **Escape key:** Handled natively by `<dialog>` — fires a `cancel` event on the element. We listen for this and emit `cancelled`. Note: `preventDefault()` on the `cancel` event is NOT reliable across browsers (Chrome/Firefox treat it as non-cancellable). Instead, we simply emit `cancelled` and let the dialog close naturally.
- **Enter key:** No explicit `(keydown.enter)` binding is needed. Native `<button>` elements fire their `click` event when Enter is pressed while focused. Since the cancel button is listed first (receives initial tab focus for safety), the user must explicitly Tab to the confirm button and press Enter to confirm. This is the safest default for destructive actions — no accidental confirms.

### Accessibility

- `aria-labelledby` on the `<dialog>` element pointing to the `<h2>` title element via an `id`
- Native `role="dialog"` and `aria-modal="true"` provided by `.showModal()`
- Focus trapping provided natively by `.showModal()`

### Module Boundaries

- Component lives in `src/app/shared/components/confirm-dialog/`
- Exported via the existing barrel at `src/app/shared/components/index.ts`
- No service needed — the component is purely presentational
- Consumers: PauseMenuComponent (T-2026-135), Settings page (T-2026-133)

## Detailed Implementation

### 1. `confirm-dialog.ts` (~55 LOC)

```typescript
import { Component, ElementRef, afterNextRender, input, output, viewChild } from '@angular/core';

@Component({
  selector: 'nx-confirm-dialog',
  template: `
    <dialog
      #dialogEl
      class="confirm-dialog"
      [class.confirm-dialog--danger]="variant() === 'danger'"
      [class.confirm-dialog--warning]="variant() === 'warning'"
      aria-labelledby="confirm-dialog-title"
      (cancel)="onCancel()">
      <h2 id="confirm-dialog-title" class="confirm-dialog__title">{{ title() }}</h2>
      <p class="confirm-dialog__message">{{ message() }}</p>
      <div class="confirm-dialog__actions">
        <button
          type="button"
          class="confirm-dialog__btn confirm-dialog__btn--cancel"
          (click)="onCancelClick()">
          {{ cancelLabel() }}
        </button>
        <button
          type="button"
          class="confirm-dialog__btn confirm-dialog__btn--confirm"
          (click)="onConfirmClick()">
          {{ confirmLabel() }}
        </button>
      </div>
    </dialog>
  `,
  styleUrl: './confirm-dialog.scss',
})
export class ConfirmDialogComponent {
  readonly title = input.required<string>();
  readonly message = input.required<string>();
  readonly confirmLabel = input<string>('Confirm');
  readonly cancelLabel = input<string>('Cancel');
  readonly variant = input<'danger' | 'warning'>('danger');

  readonly confirmed = output();
  readonly cancelled = output();

  private readonly dialogEl = viewChild.required<ElementRef<HTMLDialogElement>>('dialogEl');

  constructor() {
    afterNextRender(() => {
      this.dialogEl().nativeElement.showModal();
    });
  }

  onConfirmClick(): void {
    this.confirmed.emit();
    this.dialogEl().nativeElement.close();
  }

  onCancelClick(): void {
    this.cancelled.emit();
    this.dialogEl().nativeElement.close();
  }

  onCancel(): void {
    // Native cancel event from Escape key — dialog closes naturally
    // No preventDefault() — it is non-cancellable in Chrome/Firefox
    this.cancelled.emit();
  }
}
```

**Notes:**
- `input.required<string>()` for `title` and `message` because they must always be provided.
- `input<string>('Confirm')` etc. for optional inputs with defaults.
- `viewChild.required` uses the signal-based query API (Angular 17+).
- `afterNextRender` ensures `showModal()` is called after the element is in the DOM.
- The `cancel` event on `<dialog>` fires when the user presses Escape. The dialog closes natively; we just emit `cancelled`.
- `onCancel` does NOT call `close()` — the browser already closes the dialog on Escape.
- Cancel button is listed first (left) so it receives natural tab focus, reducing accidental destructive-action confirmation.
- No explicit Enter key binding needed — native `<button>` fires `click` on Enter. No dialog-level keydown listener.
- `aria-labelledby="confirm-dialog-title"` on the `<dialog>` ensures screen readers announce the dialog title on open.

### 3. `confirm-dialog.spec.ts` (~130 LOC)

Uses a `TestHost` pattern consistent with existing specs. The `setup()` function handles jsdom's lack of `showModal`/`close`.

**Test setup pattern:** Install jsdom polyfills on `HTMLDialogElement.prototype` BEFORE calling `createComponent`. Since polyfills are set on the prototype globally, they are in place before any component instance calls `showModal()` during `afterNextRender`. Spies are also installed on the prototype before `createComponent` so they intercept calls from any instance.

```typescript
import { Component } from '@angular/core';
import { vi } from 'vitest';
import { createComponent } from '../../../../testing/test-utils';
import { ConfirmDialogComponent } from './confirm-dialog';

@Component({
  template: `
    @if (open) {
      <nx-confirm-dialog
        [title]="title"
        [message]="message"
        [confirmLabel]="confirmLabel"
        [cancelLabel]="cancelLabel"
        [variant]="variant"
        (confirmed)="onConfirm()"
        (cancelled)="onCancel()" />
    }
  `,
  imports: [ConfirmDialogComponent],
})
class TestHost {
  open = true;
  title = 'Delete Progress';
  message = 'This action cannot be undone.';
  confirmLabel = 'Delete';
  cancelLabel = 'Keep';
  variant: 'danger' | 'warning' = 'danger';
  onConfirm = vi.fn();
  onCancel = vi.fn();
}

// Minimal TestHost without label overrides (for testing defaults)
@Component({
  template: `
    <nx-confirm-dialog
      [title]="'Reset'"
      [message]="'Are you sure?'"
      [variant]="'danger'" />
  `,
  imports: [ConfirmDialogComponent],
})
class TestHostDefaults {}
```

**Setup function:**
```typescript
async function setup(hostClass = TestHost) {
  // Install jsdom polyfills BEFORE component renders
  if (!HTMLDialogElement.prototype.showModal) {
    HTMLDialogElement.prototype.showModal = function () {
      this.setAttribute('open', '');
    };
  }
  if (!HTMLDialogElement.prototype.close) {
    HTMLDialogElement.prototype.close = function () {
      this.removeAttribute('open');
    };
  }

  const showModalSpy = vi.spyOn(HTMLDialogElement.prototype, 'showModal');
  const closeSpy = vi.spyOn(HTMLDialogElement.prototype, 'close');

  const result = await createComponent(hostClass);
  // afterNextRender fires during detectChanges (called by createComponent)
  await result.fixture.whenStable();

  const dialogEl = result.element.querySelector('dialog') as HTMLDialogElement;
  return { ...result, dialogEl, showModalSpy, closeSpy };
}
```

**Note:** Polyfills and spies are installed on `HTMLDialogElement.prototype` before `createComponent` is called, so they are in place globally before any instance's `afterNextRender` callback fires. No need for `detectChanges: false`.
