# Plan: T-2026-339 — Wire MinigameCardComponent into MinigameHubPage grid layout

## Task

MinigameHubPage (T-2026-076, completed) renders a grid of minigame cards using inline template markup. MinigameCardComponent (T-2026-187, completed) is a dedicated reusable card component with mastery stars, lock state, and level completion display. This ticket replaces the inline card rendering in MinigameHubPage with the reusable MinigameCardComponent.

Acceptance criteria:
- MinigameHubPage renders MinigameCardComponent for each registered minigame
- Card inputs populated from MinigameRegistryService (config), MasteryService (mastery), and LevelProgressionService (levels completed)
- Locked/unlocked state derived from GameProgressionService
- Card click navigates to `/minigames/:gameId` (level select)
- Grid layout maintained with responsive columns
- Unit tests for: card rendering per game, data binding from services, click navigation

## Architecture

**Approach: Replace inline card markup with `<nx-minigame-card>` component usage.**

The MinigameHubPage currently renders each card inline using a `<div class="minigame-hub__card">` with nested `<nx-locked-content>`, `<nx-mastery-stars>`, and stats markup (lines 45-60 of minigame-hub.ts). The MinigameCardComponent already encapsulates all of this: it accepts `config`, `mastery`, `levelsCompleted`, `isLocked`, and `unlockMessage` inputs, and emits `cardClicked` with the game ID.

**Key decisions:**

1. **Drop `bestScore` display.** The inline hub currently shows `Best: {{ game.bestScore }}` but the MinigameCardComponent intentionally does not have a bestScore input -- it shows only `levelsCompleted/totalLevels levels`. This is the right call: the card component was designed this way in T-2026-187, and the best-score stat is noisy for a hub overview. The `bestScore` field will be removed from the local `MinigameCardData` interface.

2. **Drop redundant imports.** MinigameHubPage currently imports `MasteryStarsComponent` and `LockedContentComponent` directly. After wiring in `MinigameCardComponent`, these are no longer needed in the hub page's imports array (they are already imported by the card component internally).

3. **Simplify card styling.** The hub's `.minigame-hub__card`, `.minigame-hub__card--unlocked`, `.minigame-hub__card-name`, `.minigame-hub__card-topic`, `.minigame-hub__card-stats` CSS classes become dead code. The card component has its own `:host` styling. The hub only needs to style the grid container and add `tabindex`/keyboard handling.

4. **Keyboard accessibility.** The inline cards currently use `[tabindex]` and `(keydown.enter)` on the wrapper div. The MinigameCardComponent does NOT set a tabindex on its host. We need to add `[attr.tabindex]` to the host bindings in MinigameCardComponent so keyboard navigation works. Alternatively, we can set tabindex on the `<nx-minigame-card>` element in the hub template. The better approach is to add it to the card component's host bindings (since any consumer would want keyboard nav), but that changes a file outside the ticket scope. Instead, we will set tabindex directly on the `<nx-minigame-card>` element in the hub template, keeping the card component unchanged. This is simpler and avoids scope creep. A follow-up ticket can add native keyboard support to MinigameCardComponent.

5. **Handle `cardClicked` output.** The card emits `cardClicked` with the `MinigameId`. The hub handles this by calling `router.navigate(['/minigames', gameId])`. The `onCardClick` method signature changes from accepting `MinigameCardData` to accepting `MinigameId`.

## Implementation Order

1. **Modify `minigame-hub.ts`** -- This is the main change:
   - Remove `MasteryStarsComponent` and `LockedContentComponent` from imports array
   - Add `MinigameCardComponent` to imports array
   - Remove `bestScore` from the `MinigameCardData` interface
   - Remove `bestScore` computation from `buildCardData()`
   - Replace the inline card `<div>` template (lines 45-60) with:
     ```html
     <nx-minigame-card
       [config]="game.config"
       [mastery]="game.masteryStars"
       [levelsCompleted]="game.levelsCompleted"
       [isLocked]="!game.isUnlocked"
       [unlockMessage]="game.unlockMessage"
       [attr.tabindex]="game.isUnlocked ? 0 : -1"
       (cardClicked)="onCardClick($event)"
       (keydown.enter)="onCardClick(game.config.id)" />
     ```
   - Simplify `onCardClick` to accept `MinigameId` instead of `MinigameCardData`:
     ```typescript
     onCardClick(gameId: MinigameId): void {
       this.router.navigate(['/minigames', gameId]);
     }
     ```
     Note: locked-state click prevention is handled by MinigameCardComponent (it does not emit `cardClicked` when locked), so the hub does not need to check `isUnlocked` itself. However, the `keydown.enter` on the hub still needs guarding since it bypasses the card's click handler. Better approach: remove the hub-level `keydown.enter` and handle it in the card. But since the card component doesn't have keydown handling, we should add a guard: `(keydown.enter)="game.isUnlocked && onCardClick(game.config.id)"`.
