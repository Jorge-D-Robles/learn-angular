# Plan: T-2026-319 — Wire registry scoreConfig.maxScore into MinigamePlayPage buildMinigameResult

## Task

`MinigamePlayPage.buildMinigameResult()` currently reads `maxScore` from `eng.config.maxScore` (engine-level, hardcoded 1000). Now that per-game `scoreConfig` exists on `MinigameConfig` (T-2026-302), the page should prefer `scoreConfig.maxScore` from the registry to prevent divergence between registry and engine config.

Acceptance criteria:
- `buildMinigameResult()` uses `registryConfig.scoreConfig?.maxScore` when available, falling back to `eng.config.maxScore`
- Unit test: score and star rating use registry scoreConfig.maxScore when it differs from engine config
- Unit test: falls back to engine config maxScore when scoreConfig is undefined

## Architecture

**The problem:** `buildMinigameResult()` (line 366-381 of `minigame-play.ts`) computes `maxScore` solely from `eng.config.maxScore`, which is the engine-level default (1000). Per-game scoring weights were added to `MinigameConfig.scoreConfig` in T-2026-302, but the play page never reads them. This means a game with `scoreConfig.maxScore = 500` would still compute star ratings against 1000.

**The fix:** A single-line change to `buildMinigameResult()` to resolve `maxScore` with a fallback chain:

```
registry scoreConfig.maxScore  -->  engine config.maxScore
```

The `gameConfig()` computed signal already exists (line 126-130) and returns the `MinigameConfig` from the registry. We just need to read `gameConfig()?.scoreConfig?.maxScore` and fall back to `eng.config.maxScore` if it is `undefined`.

**Why this approach:**
- Minimal change footprint -- one line of production code.
- The registry is the source of truth for per-game configuration. The engine config is a runtime default.
- The fallback to `eng.config.maxScore` preserves backward compatibility for any games that lack a `scoreConfig`.
- No new dependencies or services are needed; `gameConfig()` is already injected and available.
