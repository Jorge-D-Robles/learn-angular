# Plan: T-2026-508 — Wire TerminalHackFormEvaluationService into TerminalHackEngine lifecycle

## Task

Wire TerminalHackFormEvaluationService (T-2026-429) into TerminalHackEngine (T-2026-087) lifecycle, following the established P2/P3 wiring pattern (T-2026-469, T-2026-470, T-2026-471, T-2026-497). The engine currently bundles form evaluation inline alongside an optional `TerminalHackSimulationService` interface. This ticket makes the engine delegate to the real `TerminalHackFormEvaluationService` when provided, and removes the duplicated inline evaluation logic.

## Architecture

### Current State

The engine already defines a `TerminalHackSimulationService` interface (lines 137-148 of `terminal-hack.engine.ts`) with `evaluateForm()`, `runTestCases()`, and optional `reset()`. The `TerminalHackFormEvaluationService` already `implements TerminalHackSimulationService` (line 61 of `terminal-hack-evaluation.service.ts`). The engine constructor already accepts an optional `simulationService` parameter and delegates `runTestCases()` to it when present (line 370-372). It also calls `reset()` on `onLevelLoad` (line 247).

**What is already wired:**
- Constructor accepts optional `TerminalHackSimulationService`
- `onLevelLoad()` calls `_simulationService?.reset?.()`
- `runTestCases()` delegates to `_simulationService.runTestCases()` when service exists
- Existing tests verify delegation, inline fallback, and reset

**What the ticket requires but is NOT yet wired:**
1. The engine does NOT call `evaluateForm()` on the service anywhere. This ticket adds a public `evaluateFormElements()` method that delegates to the service's `evaluateForm()`.
2. Duplicated inline logic in the engine (lines 398-544: `_evaluateTestCasesInline`, `_evaluateSingleTestCase`, `_findPlacedElementByName`, `_isToolValidForFormType`, `_validateFieldValue`, `_getFieldErrors`, `_applyValidationRule`, and the duplicated constant sets) should be removed since the service now owns this.

### Design Decision: Inline Fallback for evaluateFormElements()

For `runTestCases()`, the inline fallback exists because the engine had that logic before the service was created — removing it would break 49 existing tests. For the **new** `evaluateFormElements()` method, there is no pre-existing inline code to preserve. Two options:

1. **Return `null` when no service** — simplest, no dead code, but breaks the established pattern across all engines.
2. **Provide inline fallback** — matches the pattern, but creates a new code path that duplicates the service.

**Decision: Option 1 — return `null` when no service.** Since `evaluateFormElements()` never existed in the engine, there is no legacy path to preserve. The method is an opt-in feature that requires the service. This avoids creating a new untested duplicate. The UI will call this method only when the service is present (which is always in production via `app.config.ts`).

This means:
- Existing 49 tests pass unchanged (they don't call `evaluateFormElements()`)
- New tests explicitly test both the "with service" and "without service returns null" paths
- No duplicated evaluation logic in the engine

### What This Ticket Actually Does

Since the engine already has the `TerminalHackSimulationService` interface, constructor injection, `runTestCases()` delegation, and `reset()` call, the actual remaining work is:

1. **Add `evaluateFormElements()` delegation** -- add a public `evaluateFormElements()` method on the engine that delegates to the service's `evaluateForm()`. Returns `null` when no service is provided (see design decision above).
2. **Consolidate duplicated constants** -- the engine and service both define `TEMPLATE_DRIVEN_TOOLS`, `REACTIVE_TOOLS`, `VALIDATOR_TOOLS`. Move to `terminal-hack.types.ts` as the shared source.
3. **Add wiring-specific tests** that verify delegation with the real `TerminalHackFormEvaluationService`.

### Constants Consolidation

Move the three tool constant sets (`TEMPLATE_DRIVEN_TOOLS`, `REACTIVE_TOOLS`, `VALIDATOR_TOOLS`) from both files into `terminal-hack.types.ts` as exported constants. Both the engine and service import from there. This eliminates the duplication noted in the service's comment on line 38: "Constants (copied from engine; T-2026-508 will consolidate)".

## Test Plan

### New Tests (in engine spec, new describe block: "evaluateFormElements")

1. **`evaluateFormElements() returns per-element results via service`** -- Create engine with `new TerminalHackFormEvaluationService()` (concrete type, not interface), initialize, place correct element, call `evaluateFormElements()`, verify it returns `ElementEvaluationResult[]` with `correctType: true`.

2. **`evaluateFormElements() returns null when no service provided`** -- Create engine WITHOUT service, initialize, call `evaluateFormElements()`, verify it returns `null`.

3. **`evaluateFormElements() returns all-false for missing elements`** -- Create engine with real service, initialize, do NOT place any elements, call `evaluateFormElements()`. Verify all elements report `correctType: false` and `missingValidations` contains the validation types from the spec elements (e.g., `['required']` for a spec element with a `required` validation rule).

4. **`evaluateFormElements() returns null when not Playing`** -- Call before `start()`, verify null return.

### New Tests (in engine spec, new describe block: "real service wiring")

5. **`engine with real service: runTestCases delegates to service and completes`** -- Create engine with `new TerminalHackFormEvaluationService()`, initialize, place correct element, run tests, verify Won status and score > 0. **Note:** This is distinct from the existing mock-based delegation test (section 14 of spec) — it verifies end-to-end with the real service.

6. **`engine with real service: failed tests cost a life`** -- Place wrong element, run tests via service, verify lives decrease.

7. **`engine with real service: reset propagates to service`** -- Initialize, run tests (to produce non-zero `_lastTestRunResult`), then re-initialize via `onLevelLoad`. Verify `getTestPassRate()` returns 0 after re-initialization. **Important:** The test must run tests first to distinguish reset from initial state, since `getTestPassRate()` also returns 0 before any tests run. The service variable must be typed as `TerminalHackFormEvaluationService` (concrete type, not the interface) to access `getTestPassRate()`.

8. **`engine with real service: evaluateFormElements returns per-element feedback`** -- Place a partially correct form (correct type but wrong validations), call `evaluateFormElements()`, verify mixed results.

All 49 existing engine tests must continue to pass unchanged. They create engines without the service and exercise the inline fallback path for `runTestCases()`.
