# Plan: T-2026-431 — Create TerminalHackCodePanelComponent for target form and code editor split view

## Task

Extract the split-panel layout (left: target form spec preview, right: code editor) from `TerminalHackComponent` into a standalone `TerminalHackCodePanelComponent`. The component provides the retro terminal aesthetic (green-on-black, scanlines) and handles target spec rendering, code editor display, tool availability hints, and code change emission.

## Architecture

### Design Decision: Pure Presentation Sub-component

This component follows the established P2 sub-component extraction pattern used by:
- `BindingTypeSelectorComponent` (wire-protocol/binding-type-selector/)
- `FlowCommanderGateConfigComponent` (flow-commander/gate-config/)
- `SignalCorpsTowerConfigComponent` (signal-corps/tower-config/)

All of these are pure presentation components that:
- Live in a subdirectory of their parent minigame
- Use `input()` / `input.required()` for data flow in, `output()` for events out
- Have their own SCSS file with BEM naming
- Are tested via a `TestHost` wrapper component
- Are exported from the parent minigame's barrel `index.ts`

### Input/Output Contract

The ticket specifies `AvailableFormTool[]` as an input type, but this type does not exist in the codebase. The existing type for form tools is `FormToolType` from `terminal-hack.types.ts`. Since `availableElements` in the engine is typed as `readonly FormToolType[]`, the component will use `FormToolType[]` directly. This aligns with how `BindingTypeSelectorComponent` uses `WireType[]` rather than inventing a wrapper type.

**Note:** The ticket AC should be read as `availableTools (FormToolType[])` — this is a type name correction, not a scope change.

**Output naming:** The output is named `codeChange` (not `codeChanged` as in the ticket AC) to match `CodeEditorComponent`'s own output naming convention. The downstream wiring ticket T-2026-511 should use `(codeChange)="..."` when binding. This is an intentional deviation from the AC for consistency with the existing codebase convention.

### State Management: initialCode

The `initialCode` input seeds an internal writable signal via an `effect`, following the established pattern from `RouteEditorComponent` (corridor-runner/route-editor). The internal signal is what binds to `CodeEditorComponent`'s `[code]` input. This prevents parent re-renders from overwriting user edits. A `_seeded` flag ensures one-shot seeding.

```
initialCode = input<string>('');
private _code = signal('');
private _seeded = false;

constructor() {
  effect(() => {
    const code = this.initialCode();
    if (!this._seeded) {
      this._seeded = true;
      this._code.set(code);
    }
  });
}
```

### ALL_FORM_TOOL_TYPES Constant

The component defines a local `ALL_FORM_TOOL_TYPES: readonly FormToolType[]` constant enumerating all values of the `FormToolType` union (matching how `BindingTypeSelectorComponent` uses `WIRE_TYPE_OPTIONS`). This drives the tool palette — available tools are enabled, the rest are dimmed.

### Why a Standalone Sub-component

The parent `TerminalHackComponent` currently renders the left panel (target spec), palette, and code editor inline across ~70 lines of HTML. Extracting the code panel:
1. Reduces the parent template to a single `<app-terminal-hack-code-panel>` tag for this section
2. Encapsulates the retro terminal aesthetic (scanlines, cursor blink) in a reusable unit
3. Makes the wiring ticket (T-2026-511) straightforward -- just bind inputs/outputs
4. Keeps the parent focused on orchestrating engine interactions

### Panel Layout

- **Left sub-panel**: Read-only target form spec with required fields highlighted via an orange left-border on elements that have a `required` validation rule. Shows element label, type badge, and validation tags.
- **Right sub-panel**: `CodeEditorComponent` in editable mode (`[readOnly]="false"`). This is a NEW editable code-writing surface — distinct from the parent's existing read-only synthesized code preview. The internal `_code` signal (seeded from `initialCode`) binds to `[code]`. The `codeChange` output from `nx-code-editor` updates `_code` and forwards as the `codeChange` output of this component.
- **Tool palette**: Sits above the code editor with `role="toolbar"` and individual tool buttons with `aria-label`. Available tools are rendered as enabled buttons. Tools NOT in `availableTools` are shown as dimmed (`aria-disabled="true"`, reduced opacity), providing a "what's available at this level" hint.

### Retro Terminal Aesthetic

The parent `TerminalHackComponent` already implements the retro terminal aesthetic (green-on-black, scanlines, mono font). This sub-component **redeclares** the same SCSS color variables locally (`$tg`, `$bg`, `$tb`, `$glow`) since they are local to the parent's SCSS file, not globally available. This is acceptable duplication — there is no shared SCSS tokens file in the codebase.

- A cursor-blink animation implemented as a real DOM element `<span class="code-panel__cursor">_</span>` (not a pseudo-element, so it is testable via `querySelector`)
- The scanline overlay is handled by the parent and covers the entire game area, so the sub-component does NOT duplicate it
- `prefers-reduced-motion` media query disables the blink animation

## Test Plan

Tests use the TestHost wrapper pattern (matching BindingTypeSelectorComponent and FlowCommanderGateConfigComponent). All tests use `fixture.nativeElement` DOM queries, not `debugElement`.

### Test Cases

1. **Rendering: target spec elements** -- Given a `targetSpec` with 2 elements, expect 2 `.code-panel__spec-item` elements rendered with correct labels and type badges.

2. **Rendering: required fields highlighted** -- Given a `targetSpec` with one element having `required` validation and one without, expect the required element to have the `.code-panel__spec-item--required` CSS class.

3. **Rendering: code editor present** -- Expect `nx-code-editor` element to be present in the DOM.

4. **Rendering: code editor receives initialCode** -- Given `initialCode` of `'// starter'`, expect the `nx-code-editor` to have `[code]` bound to that value.

5. **Rendering: tool palette buttons** -- Given `availableTools` of `['FormControl', 'Validators.required']`, expect 2 enabled tool buttons with correct text.

6. **Rendering: all FormToolType values shown with unavailable dimmed** -- Given `availableTools` of `['FormControl']`, expect `FormControl` button enabled and all other `ALL_FORM_TOOL_TYPES` entries rendered as disabled buttons with `.code-panel__tool-btn--dimmed` class. `ALL_FORM_TOOL_TYPES` is a constant defined in `code-panel.ts`.

7. **Output: codeChange emitted on editor input** -- Trigger `codeChange` output on the `nx-code-editor` child component instance, expect the TestHost's `onCodeChange` spy to receive the new code string.

8. **Retro theme: host has code-panel class** -- Assert that the root element has the `.code-panel` CSS class present (DOM class check, not computed style).

9. **Retro theme: cursor blink element present** -- Expect a real DOM element `.code-panel__cursor` (a `<span>`) to exist in the rendered output.

10. **ARIA: toolbar role on palette** -- Expect the tool palette container to have `role="toolbar"` and each tool button to have an `aria-label`.

11. **Edge case: empty targetSpec** -- Given a `targetSpec` with no elements, expect 0 `.code-panel__spec-item` elements and no errors.

12. **Edge case: empty initialCode** -- Given empty string `initialCode`, expect `nx-code-editor` to render without errors.

13. **Edge case: empty availableTools** -- Given empty `availableTools`, expect all `ALL_FORM_TOOL_TYPES` buttons rendered as dimmed (none enabled).
