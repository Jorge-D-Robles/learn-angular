### Type Definitions

The Power Grid types file (`power-grid.types.ts`) does not exist yet -- the barrel at `src/app/features/minigames/power-grid/index.ts` exports nothing. T-2026-270 (the dedicated types ticket) is still `todo`. However, T-2026-091's acceptance criteria explicitly require `PowerGridLevelData` with `services[], components[], validConnections[], scopeRules[]`. Following the established pattern (Corridor Runner types in `corridor-runner.types.ts`, Terminal Hack types in `terminal-hack.types.ts`), this ticket must create `power-grid.types.ts` with the minimum types needed by the level data.

**Decision: Create `power-grid.types.ts` with the types required by level data only.** T-2026-270 can later add additional types needed by the engine and UI (like `ShortCircuit`, `PowerConnection.isCorrect`, full validation logic types). The types created here are a strict subset of what T-2026-270 specifies -- no conflicts, just partial fulfillment.

Types to define in `power-grid.types.ts`:
All interface fields use `readonly` modifiers (matching Terminal Hack and Corridor Runner conventions).

- `InjectionScope` -- string union: `'root' | 'component' | 'hierarchical'` (3 values only — these are actual Angular DI scopes. `'root'` = singleton via `providedIn: 'root'`, `'component'` = provided at component level, `'hierarchical'` = provided at a parent component, inherited by children)
- `ProviderType` -- string union: `'class' | 'factory' | 'value' | 'existing'` (how the service is provided — `useClass`, `useFactory`, `useValue`, `useExisting`. Basic levels default to `'class'`; Advanced levels use factory/value/existing)
- `ServiceNode` -- `{ readonly id, readonly name, readonly type, readonly providedIn, readonly providerType?, readonly kind?, readonly dependsOn?, readonly methods?, readonly stateful? }`. `kind?: 'class' | 'token'` distinguishes class-based services from `InjectionToken`s (levels 11, 15). `dependsOn?: readonly string[]` models service-to-service injection (level 10). `providerType` defaults to `'class'` when omitted.
- `ComponentNode` -- `{ readonly id, readonly name, readonly requiredInjections, readonly providers? }` (providers is optional for basic levels)
- `ValidConnection` -- `{ readonly serviceId, readonly componentId, readonly scope }` (the answer key — which service connects to which component at what scope. Note: T-2026-270 will add `PowerConnection` with `isCorrect` for runtime use; `ValidConnection` is the static answer-key counterpart)
- `ScopeRule` -- `{ readonly serviceId, readonly allowedScopes, readonly defaultScope }` (constrains what scopes are valid for a service)
- `PowerGridLevelData` -- `{ readonly services, readonly components, readonly validConnections, readonly scopeRules }` (the top-level type parameterizing `LevelDefinition<T>`)

**Why `InjectionScope` and `ProviderType` are separate:** The original plan conflated scope and provider type (e.g., `'factory'` as a scope). In Angular, `useFactory` is a provider type, not a scope. Separating them prevents confusion when T-2026-270 builds the engine. T-2026-270's BACKLOG entry should be updated to note this ticket pre-creates these types.

**Why a string union instead of enum for `InjectionScope`:** The level data file uses string literals extensively. A string union keeps the data file clean (`scope: 'root'` vs `scope: InjectionScope.Root`) and follows the pattern used by other types (e.g., `FormElementType` in Terminal Hack is a string union, not an enum). T-2026-270 can add the enum separately if needed by the engine/UI.

### Level Design

The 18 levels are defined in the spec at `docs/minigames/07-power-grid.md`. Each level introduces a specific Angular DI concept from chapters 18-19 of the curriculum. The levels map to:

**Basic Tier (6 levels):** Core DI -- `@Injectable`, `providedIn: 'root'`, `inject()`, service state, service methods.
- Level 1: 1 service, 1 component (simplest possible DI)
- Level 2: 1 service, 3 components (shared service)
- Level 3: 3 services, 3 components (1:1 mapping)
- Level 4: Same as level 3 but testing `inject()` function syntax
- Level 5: 2 services with state, 3 components sharing state
- Level 6: 3 services with methods, 4 components calling methods

**Intermediate Tier (6 levels):** Scoping -- component-level providers, hierarchical injection, multiple instances, service-to-service, InjectionToken, mixed.
- Level 7: 1 service, 2 components (one uses component-level provider)
- Level 8: 1 service, 3 hierarchical components (parent provides, children inherit)
- Level 9: 1 service class, 3 components, 2 different instances via scoping
- Level 10: 3 services (one depends on another), 3 components
- Level 11: 2 services + 1 InjectionToken, 3 components
- Level 12: Mix of all scoping patterns -- 4 services, 5 components

**Advanced Tier (5 levels):** Factory/value/multi providers, optional injection, full design.
- Level 13: 2 services (one with useFactory), 3 components
- Level 14: 3 services (useValue, useExisting aliases), 3 components
- Level 15: 1 token with 3 multi-provider implementations, 3 components
- Level 16: 3 services (1 optional), 4 components
- Level 17: 5 services, 6 components -- design from scratch

**Boss Level (1 level):** "Grid Overhaul" -- 8 services, 12 components, all DI patterns combined. ParTime: 300 seconds.

### Naming Conventions

- Level IDs: `pg-basic-01` through `pg-basic-06`, `pg-intermediate-01` through `pg-intermediate-06`, `pg-advanced-01` through `pg-advanced-05`, `pg-boss-01`
- Game ID: `'power-grid'` (matches `MinigameId` union)
- Service IDs: `{levelPrefix}-svc-{n}` (e.g., `pg-b01-svc-1`)
- Component IDs: `{levelPrefix}-cmp-{n}` (e.g., `pg-b01-cmp-1`)

- Components: `BridgeModule`, `MedBayModule`, `EngineeringModule`, `CargoModule`, `SensorModule`, `ReactorModule`, `HabitatModule`, `CommsModule`, `ArmoryModule`, `ScienceLabModule`, `HangarModule`, `CommandModule`
