# Plan: T-2026-509 — Create TerminalHackLivePreviewComponent for real-time form preview panel

## Task

Terminal Hack spec describes "Live preview updates as the player types" and "Left panel: target form preview; Right panel: form code editor." The live preview panel shows the player's form taking shape in real time. Extracting the live preview into a standalone sub-component follows the established pattern (like `TerminalHackCodePanelComponent` and `FlowCommanderGateConfigComponent`) and keeps the main component manageable.

Acceptance criteria:
- `TerminalHackLivePreviewComponent` at `src/app/features/minigames/terminal-hack/live-preview/live-preview.ts`
- Input: `formElements` (PlayerFormElement[]) -- the player's current form structure
- Input: `targetSpec` (TargetFormSpec) -- the target form for reference comparison
- Renders a visual representation of the form the player is building (labels, input fields, buttons)
- Updates reactively as the player adds/modifies form elements
- Visual indicators: correctly-placed elements glow green, missing elements shown as dashed placeholders
- Terminal aesthetic: green-on-black rendering matching the retro theme
- Output: `elementClicked` event with elementId for editing
- Exported from terminal-hack barrel
- Unit tests for: form element rendering, reactive updates, correct/missing indicators, terminal theme applied

## Architecture

### Component Design

The `TerminalHackLivePreviewComponent` is a **presentational sub-component** that receives data via signal inputs and emits events via outputs. It has no dependencies on the engine or any services -- the parent `TerminalHackComponent` will bridge the engine signals to this component's inputs.

### Data Flow

```
TerminalHackEngine.placedElements() --> TerminalHackComponent --> [formElements] --> LivePreviewComponent
TerminalHackEngine.targetFormSpec() --> TerminalHackComponent --> [targetSpec]   --> LivePreviewComponent
                                                              <-- (elementClicked) <-- LivePreviewComponent
```

### Key Design Decisions

1. **Iterate over `targetSpec.elements`, not `formElements`.** The preview should always show all target form slots (both filled and empty). For each target element, we check whether a matching `PlayerFormElement` exists in the `formElements` array. This gives us the "dashed placeholder for missing, green glow for correct" behavior described in the acceptance criteria.

2. **Matching logic uses `elementId`.** Each `FormElementSpec` in the target has an `id` field. Each `PlayerFormElement` also has an `elementId` field. We match by ID. Additionally, we check whether the player placed the correct `elementType` to determine if an element is "correctly-placed" (glow green) vs "placed but wrong" (no glow).

3. **Array input, not Map.** The ticket specifies `formElements` as `PlayerFormElement[]`. The engine exposes `placedElements` as `ReadonlyMap<string, PlayerFormElement>`, but the parent component will convert it to an array (or the `FormPreview.elements` array is already available). Using an array keeps the component interface simple. Internally, the component builds a lookup Map via `computed()` for O(1) matching.

4. **Inline template.** Following the code-panel pattern, the component uses an inline template since the HTML is modest in size. The SCSS is in a separate file for reuse of the terminal theme variables.

5. **No service dependencies.** Pure presentational component. All comparison logic is done via computed signals inside the component.

7. **`PreviewSlot` interface** — defined locally in the component file:
   ```typescript
   interface PreviewSlot {
     spec: FormElementSpec;
     placed: PlayerFormElement | undefined;
     status: 'missing' | 'correct' | 'incorrect';
   }
   ```
   The `status` field drives the BEM modifier class (`--missing`, `--correct`, `--incorrect`).

8. **Wiring to parent is out of scope.** This ticket creates the standalone component. Wiring `elementClicked` to the parent's `onSelectSlot()` is done in a separate wiring ticket (T-2026-512).

6. **Selector:** `app-terminal-hack-live-preview` following the `app-` prefix convention for feature components.

### Visual Rendering Strategy

Each target element renders as a "form field preview":
- **Label** above the field (from `FormElementSpec.label`)
- **Visual input mock** styled according to `elementType` (text input, select dropdown, checkbox, etc.)
- **Status indicator:**
  - Missing (no player element placed): dashed border, dim opacity, "Missing" label
  - Placed correctly (matching elementType): solid green border with glow
  - Placed incorrectly (wrong elementType): solid orange/warning border
- **Clickable** -- clicking emits `elementClicked` with the element's ID

The form name and form type appear as a header above the elements.

### Test Cases (14 tests)

1. **Rendering: form name header** -- Should display the `targetSpec.formName` in a header element.
2. **Rendering: form type badge** -- Should display the `targetSpec.formType` in a badge element.
3. **Rendering: all target elements shown** -- Should render one preview slot per target element (2 for TWO_ELEMENT_SPEC).
4. **Rendering: element labels displayed** -- Each preview slot shows the element's label text ("Username", "Email").
5. **Rendering: element type shown** -- Each preview slot shows the element type ("text", "email").
6. **Missing elements: dashed placeholder** -- When `formElements` is empty, all slots have the `--missing` CSS modifier class (dashed border placeholder).
7. **Correct placement: green glow** -- When a `PlayerFormElement` with matching ID and correct `elementType` is in `formElements`, the slot has the `--correct` CSS modifier class.
8. **Incorrect placement: warning indicator** -- When a `PlayerFormElement` has matching ID but wrong `elementType`, the slot has the `--incorrect` CSS modifier class (not `--correct`).
9. **Reactive updates** -- After adding a formElement at runtime (changing the input), the DOM updates to show the slot as filled.
10. **Output: elementClicked emitted on click** -- Clicking a preview slot emits `elementClicked` with the element's ID string.
11. **Output: elementClicked emitted on Enter key** -- Dispatch `new KeyboardEvent('keydown', { key: 'Enter' })` on a focused slot element, expect `elementClicked` to emit with the element's ID (keyboard accessibility).
12. **Edge case: empty targetSpec** -- When targetSpec has 0 elements, renders 0 slots without errors.
13. **Edge case: empty formElements** -- All slots render as missing when formElements array is empty.
14. **Terminal theme: host has live-preview class** -- The root `.live-preview` CSS class is present on the component.
