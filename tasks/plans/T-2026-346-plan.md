# Plan: T-2026-346 — Create E2E smoke tests for P2 page components

## Task

Create Playwright E2E smoke tests that verify each P2 page component (Dashboard, Profile, Settings, Campaign, MinigameHub) renders its key sections correctly at both mobile (768px) and desktop (1280px) viewports. These tests complement the existing `app.spec.ts` (basic routing), `dashboard-responsive.spec.ts` (layout-specific), and `minigame-shell.spec.ts` (minigame play page) E2E tests by asserting that page-level content actually renders with real data.

## Architecture

**Approach:** A single spec file `e2e/pages.spec.ts` with one `test.describe` block per page. Each page gets two tests — one at desktop viewport (1280x720) and one at mobile viewport (768x1024). The tests use Playwright's `getByRole`, `getByText`, and CSS class locators to verify that key structural sections are present and visible.

**Why this approach over per-page spec files:** The ticket explicitly requests a single `e2e/pages.spec.ts` file. Smoke tests are intentionally lightweight — they verify that the page renders its major sections, not that every interactive feature works. Keeping them in one file makes it easy to run the full smoke suite and see which pages are broken at a glance.

**Why both viewports:** The ticket requires responsive layout verification at mobile (768px) and desktop (1280px). At mobile, some layout changes occur (e.g., dashboard cards stack vertically, stats grid collapses). Testing both ensures the responsive CSS does not hide critical sections.

**Data dependency:** All pages work with default/initial state (0 XP, no completed missions). This is by design — the E2E server starts fresh. The tests verify that the pages render correctly in this initial state, which is the most common first-time-user scenario.

**What NOT to test here:** Interactive behavior (clicking buttons, navigating), detailed content accuracy (exact XP numbers), or visual regression. Those belong in dedicated E2E or unit tests.

## Key Implementation Details

### Locator strategy

Follow the existing codebase pattern:
- Use `page.getByRole('heading', { name: '...' })` for headings
- Use `page.locator('.css-class')` for structural sections (matching `dashboard-responsive.spec.ts` pattern)
- Use `page.locator('nx-component-name')` or `page.locator('app-component-name')` for Angular components
- Use `page.getByRole('button', { name: '...' })` for buttons
- Use `page.getByText(...)` for specific text content

### Viewport handling

Use `page.setViewportSize()` at the start of each test (matching `dashboard-responsive.spec.ts` pattern) rather than Playwright projects, since the default project is Desktop Chrome. For desktop tests, explicitly set `{ width: 1280, height: 720 }` to be deterministic.

### Wait strategy

After `page.goto(url)`, wait for the page heading to be visible before checking other elements. This ensures the component has rendered and Angular change detection has run. Example:
```typescript
await page.goto('/profile');
await expect(page.getByRole('heading', { name: 'Profile' })).toBeVisible();
```

### Settings page specifics

The sound toggle uses `aria-pressed` attribute. Verify the toggle exists with:
```typescript
await expect(page.locator('[aria-pressed]').first()).toBeVisible();
```

The select elements for animation speed and theme can be located by their IDs (`#animation-speed`, `#theme-select`).

The "Reset All Progress" button text is unique enough for `getByRole('button', { name: 'Reset All Progress' })`.

### Campaign page specifics

The campaign renders phases from `CURRICULUM` data. In initial state, all 34 missions exist across 6 phases. Use `page.locator('.campaign__phase')` and assert count `>= 6`. Check for mission cards with `page.locator('nx-mission-card')`.

### MinigameHub page specifics

All 12 minigames are registered. In initial state, most are locked (only first unlocked). The grid renders all of them regardless of lock state. Assert `page.locator('nx-minigame-card')` has count `>= 12`.

   ```bash
   npx playwright test e2e/pages.spec.ts --headed
   ```
