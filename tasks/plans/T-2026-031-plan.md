# Plan: T-2026-031 — Create code editor component wrapper

## Task

Create a reusable code editor/display component for minigames that require showing or editing code. Uses a lightweight custom syntax highlighting approach (no external editor libraries) to keep bundle size small.

## Architecture

### Decision: Custom syntax highlighter over Monaco/CodeMirror

Monaco (~3MB) and CodeMirror (~150KB) are overkill for this use case. The code editor is used for:
1. **Story missions** — read-only code display with line highlights (no editing needed)
2. **Minigames** — small code snippets (5-30 lines) where players edit specific parts

A custom solution using regex-based tokenization and `<span>` elements is the right choice because:
- Zero additional bundle size
- Full control over the Nexus Station dark theme
- Sufficient for the syntax highlighting needs (keywords, strings, comments, decorators)
- If more sophisticated editing is needed later, we can swap in CodeMirror without changing the component API

### Component Design

```
CodeEditorComponent (nx-code-editor)
  Input:  code: string           — the source code to display/edit
  Input:  language: string       — 'typescript' | 'html' (default: 'typescript')
  Input:  readOnly: boolean      — if true, no editing allowed (default: false)
  Input:  highlightLines: number[] — 1-based line numbers to highlight
  Output: codeChange: string     — emitted when the user edits code
```

Note: Input is named `readOnly` (camelCase) to avoid ambiguity with TypeScript's `readonly` modifier keyword. Template binding: `[readOnly]="true"`. The ticket AC lists `readonly` — this `readOnly` naming is an intentional deviation; the AC checkbox for `readonly` is satisfied by `readOnly`.

### Syntax Highlighting Strategy — Structured Token Arrays (not innerHTML)

The tokenizer returns a structured array of token objects, NOT an HTML string. This avoids `innerHTML` and `DomSanitizer` entirely — the template renders tokens via `@for` loops, which is the idiomatic Angular approach.

```typescript
export interface Token {
  readonly text: string;
  readonly type: TokenType;
}

export type TokenType = 'keyword' | 'string' | 'comment' | 'decorator' | 'number' | 'tag' | 'attr' | 'punctuation' | 'text';

export function tokenize(code: string, language: string): Token[]
```

The tokenizer processes the **full code string** at once (not line-by-line) to correctly handle multi-line constructs like `/* ... */` comments. The component's computed signal then splits the resulting token array into lines.

For unknown language values, the tokenizer returns the entire code as a single `text` token (no highlighting, no errors).

### Editing Strategy — Dual-Layer with Scroll Sync

For editable mode, a `<textarea>` is overlaid on the highlighted `<pre><code>` display.

**Scroll sync approach:** The wrapper `<div class="code-editor">` has `position: relative` and a fixed `max-height`. The `<textarea>` has `overflow: auto` and is the scroll source. The `<pre>` has `overflow: hidden`. A `(scroll)` event handler on the textarea syncs `pre.scrollTop = textarea.scrollTop` and `pre.scrollLeft = textarea.scrollLeft` via a `ViewChild` reference: `preRef = viewChild.required<ElementRef>('codeDisplay')` with `#codeDisplay` on the `<pre>` element.

For readonly mode, the textarea is not rendered at all.

### Line Highlighting

The component's computed signal groups tokens into lines by splitting on `\n` characters within the token array. Each line gets a `highlighted` boolean based on whether its 1-based index appears in `highlightLines`.

## Detailed File Specifications

### `syntax-highlight.ts`

Exports:

```typescript
export interface Token {
  readonly text: string;
  readonly type: TokenType;
}

export type TokenType = 'keyword' | 'string' | 'comment' | 'decorator' | 'number' | 'tag' | 'attr' | 'punctuation' | 'text';

export function tokenize(code: string, language: string): Token[]
```

**Tokenization approach:** Scan left-to-right through the raw code string. At each position, test regex patterns in priority order. When a pattern matches at the current position, push a `Token` with the matched text and type, then advance past the match. Characters that don't match any pattern are accumulated as `text` tokens.

HTML-escaping is NOT done by the tokenizer — since we use structured tokens rendered via `@for`, Angular's template binding handles escaping automatically.

**TypeScript token patterns (priority order):**
1. Single-line comments: `//.*$` (multiline flag)
2. Multi-line comments: `/\*[\s\S]*?\*/`
3. Template literals: `` `[^`]*` ``
4. Double-quoted strings: `"(?:[^"\\]|\\.)*"`
5. Single-quoted strings: `'(?:[^'\\]|\\.)*'`
6. Decorators: `@[A-Za-z]+`
7. Keywords: `\b(import|export|from|const|let|var|function|return|if|else|for|while|class|interface|type|extends|implements|new|this|true|false|null|undefined|async|await|readonly|static|private|protected|public|abstract|enum|namespace|module|declare|of|in|as|is|keyof|typeof|void|never|any|unknown|string|number|boolean|object|symbol|bigint)\b`
8. Numbers: `\b\d+\.?\d*\b`

**HTML token patterns (priority order):**
1. Comments: `<!--[\s\S]*?-->`
2. Double-quoted strings: `"(?:[^"\\]|\\.)*"`
3. Single-quoted strings: `'(?:[^'\\]|\\.)*'`
4. Tags: `<\/?[a-zA-Z][\w-]*` and `\/?>` (opening/closing)
5. Attributes: `\b[a-zA-Z][\w-]*(?==)` (word followed by `=`)
6. Interpolation: `\{\{.*?\}\}` (Angular template expressions) — maps to `keyword` token type (reuses existing type, no new TokenType needed)

**Unknown language:** Return the entire code as `[{ text: code, type: 'text' }]`.

### `code-editor.ts`

Angular 21 standalone component using signal-based inputs. Uses **inline template** (matching `station-card.ts` pattern):

```typescript
@Component({
  selector: 'nx-code-editor',
  template: `...`,
  styleUrl: './code-editor.scss',
})
```

**Inputs (signal-based):**
- `code = input<string>('')`
- `language = input<string>('typescript')`
- `readOnly = input<boolean>(false)` — camelCase to avoid TypeScript `readonly` keyword ambiguity
- `highlightLines = input<number[]>([])`

**Outputs:**
- `codeChange = output<string>()`

**Computed signals:**
- `lines = computed(() => ...)` — calls `tokenize(this.code(), this.language())` on the full code string, then groups the resulting tokens into lines by splitting on `\n` within token text. Tokens containing `\n` are split at each newline boundary into two partial tokens (text before and after the `\n`). Returns `{ tokens: Token[]; highlighted: boolean }[]`. Trailing newlines produce an empty final line (consistent with `String.split('\n')` behavior).

**Template structure:**
```html
<div class="code-editor">
  <pre class="code-display"><code>
    @for (line of lines(); track $index) {
      <span class="code-line" [class.code-line--highlighted]="line.highlighted">
        @for (token of line.tokens; track $index) {
          <span [class]="'token-' + token.type">{{ token.text }}</span>
        }
      </span>
    }
  </code></pre>
  @if (!readOnly()) {
    <textarea
      class="code-textarea"
      [value]="code()"
      (input)="onInput($event)"
      (scroll)="onScroll($event)"
      spellcheck="false"
      autocomplete="off"
    ></textarea>
  }
</div>
```

**Methods:**
- `onInput(event: Event)` — extracts value, emits `codeChange.emit(value)`. No debounce (parent is responsible for rate-limiting if needed).
- `onScroll(event: Event)` — syncs textarea scroll position to the `<pre>` element via `ViewChild`.
