# Plan: T-2026-025 -- Integrate persistence with GameStateService auto-save

## Task

Wire up `StatePersistenceService` to `GameStateService` so state is automatically saved when it changes and restored on app startup.

Acceptance criteria:
- `GameStateService` loads saved state from localStorage on initialization
- State is auto-saved when any state signal changes (using `effect()`)
- Debounce saves to avoid excessive writes (at most once per 500ms)
- If no saved state exists, initializes with defaults (Cadet rank, 0 XP)
- Unit tests verify: load on init, auto-save on mutation, debounce behavior

## Architecture

### Approach: Inject persistence into GameStateService

`GameStateService` will inject `StatePersistenceService` and handle both loading (in constructor) and auto-saving (via `effect()` with manual debounce). This keeps `StatePersistenceService` generic/reusable and puts the integration logic where it belongs -- in the state service that owns the data.

### State snapshot

Define a `GameStateSnapshot` interface representing the serializable shape of the game state:

```ts
interface GameStateSnapshot {
  playerName: string;
  totalXp: number;
}
```

This is stored under a single localStorage key: `game-state` (namespaced to `nexus-station:game-state` by `StatePersistenceService`).

### Why a single key (not per-signal keys)?

Saving the entire state as one object ensures atomicity -- we never get into a state where `totalXp` was saved but `playerName` was not. It also simplifies the load path (one `load()` call) and the debounce logic (one timer, not N timers).

### Load on init

In the constructor, call `this.persistence.load<GameStateSnapshot>('game-state')`. If the result is non-null AND has the expected shape, apply it to the private signals. If null or malformed, keep the defaults (empty string name, 0 XP). Use a simple runtime type check (not a schema validator) since this is internal data.

### Debounced auto-save via effect()

Use Angular's `effect()` to watch all public signals (`playerName()`, `totalXp()`). Inside the effect callback, clear any pending `setTimeout` and schedule a new one at 500ms. When the timeout fires, call `this.persistence.save('game-state', snapshot)`.

The effect itself reads the signal values (registering them as dependencies), captures them in a local variable, then passes them to the debounced save. This ensures the effect re-runs on every signal change but the actual write is throttled.

We need to use `untracked()` for the save call itself since `this.persistence.save()` should not create additional signal subscriptions.

A class field `private _saveTimeout: ReturnType<typeof setTimeout> | null` manages the debounce timer. On `DestroyRef`, we clear the timer to prevent saves after the service is destroyed.

### resetState enhancement

The existing `resetState()` already sets signals to defaults. With auto-save wired up, this naturally triggers the effect, which will debounce-save the zeroed state. We also add `this.persistence.clear('game-state')` for an immediate clear (so if the app crashes before the debounced save fires, there is no stale data). This replaces the TODO comment in the existing code.

### Why not use StatePersistenceService.autoSave()?

The existing `autoSave()` method watches a single signal without debounce. This ticket requires debounce and needs to watch multiple signals atomically. Building a custom `effect()` in `GameStateService` is more appropriate than extending `autoSave()` with debounce/multi-signal support (which would change the completed T-2026-024 API).

## Detailed Implementation

### game-state.service.ts changes

1. Add imports: `inject`, `effect`, `DestroyRef`, `untracked` from `@angular/core`; `StatePersistenceService` from `../persistence/state-persistence.service`.

2. Define `GameStateSnapshot` interface (exported, above the class):
   ```ts
   export interface GameStateSnapshot {
     playerName: string;
     totalXp: number;
   }
   ```

3. Add constant for the storage key:
   ```ts
   const GAME_STATE_KEY = 'game-state';
   ```

4. Inside the class, add injected dependencies:
   ```ts
   private readonly persistence = inject(StatePersistenceService);
   private readonly destroyRef = inject(DestroyRef);
   ```
   Note: `Injector` is NOT needed — `GameStateService` is `providedIn: 'root'` and its constructor runs inside Angular's injection context, so `effect()` called from the constructor works without an explicit injector.

5. Add debounce timer field:
   ```ts
   private _saveTimeout: ReturnType<typeof setTimeout> | null = null;
   ```

6. Add debounce constant:
   ```ts
   static readonly SAVE_DEBOUNCE_MS = 500;
   ```
   Making it `static readonly` so tests can reference it and it is clear this is a configuration value.

7. Add constructor that:
   - Calls `this._setupAutoSave()` FIRST to wire up the effect (ensures no signal mutation during `_loadState()` is missed — both orderings produce a debounced save on startup since loading triggers signal mutations which trigger the effect, but this order guarantees the effect captures all changes)
   - Then calls `this._loadState()` to restore from localStorage
   - Registers a cleanup callback on `DestroyRef` to clear the timeout

8. Add `_loadState()` private method:
   ```ts
   private _loadState(): void {
     const saved = this.persistence.load<GameStateSnapshot>(GAME_STATE_KEY);
     if (saved !== null && typeof saved === 'object' && !Array.isArray(saved)) {
       if (typeof saved.playerName === 'string') {
         this._playerName.set(saved.playerName);
       }
       if (typeof saved.totalXp === 'number' && saved.totalXp >= 0) {
         this._totalXp.set(saved.totalXp);
       }
     }
   }
   ```
   Defensive type checks so corrupted/partial data does not crash the app.

9. Add `_setupAutoSave()` private method:
   ```ts
   private _setupAutoSave(): void {
     effect(() => {
       const snapshot: GameStateSnapshot = {
         playerName: this.playerName(),
         totalXp: this.totalXp(),
       };
       untracked(() => this._debouncedSave(snapshot));
     });
   }
   ```

10. Add `_debouncedSave()` private method:
    ```ts
    private _debouncedSave(snapshot: GameStateSnapshot): void {
      if (this._saveTimeout !== null) {
        clearTimeout(this._saveTimeout);
      }
      this._saveTimeout = setTimeout(() => {
        this.persistence.save(GAME_STATE_KEY, snapshot);
        this._saveTimeout = null;
      }, GameStateService.SAVE_DEBOUNCE_MS);
    }
    ```

11. Update `resetState()`:
    ```ts
    resetState(): void {
      this._playerName.set('');
      this._totalXp.set(0);
      this.persistence.clear(GAME_STATE_KEY);
    }
    ```
    Remove the old TODO comment. The `clear()` call immediately removes persisted data. The signal changes will also trigger the auto-save effect, which will debounce-save the zeroed state (harmless redundancy).

**Test setup pattern:**

```ts
describe('persistence integration', () => {
  let fakeStorage: Storage;
  let originalLocalStorage: Storage;

  function createFakeStorage(): Storage { /* same as persistence spec */ }

  function seedState(snapshot: Partial<GameStateSnapshot>): void {
    const full = { playerName: '', totalXp: 0, ...snapshot };
    fakeStorage.setItem('nexus-station:game-state', JSON.stringify(full));
  }

  beforeEach(() => {
    fakeStorage = createFakeStorage();
    originalLocalStorage = window.localStorage;
    Object.defineProperty(window, 'localStorage', {

    });
    TestBed.resetTestingModule();
    TestBed.configureTestingModule({});
  });

  afterEach(() => {
    Object.defineProperty(window, 'localStorage', {
      value: originalLocalStorage, writable: true, configurable: true,
    });
    vi.useRealTimers();
  });

  // ... tests here
});
```

**IMPORTANT test setup notes:**

1. **Existing tests must also use fake localStorage.** Since the constructor now calls `effect()` which triggers auto-save, ALL tests (even the original ones) need a fake localStorage in `beforeEach` to prevent pollution of real localStorage. Move the `createFakeStorage` and `Object.defineProperty` setup into the top-level `beforeEach` for all `GameStateService` tests.

2. **Persistence tests must inject the service individually per test** (NOT in `beforeEach`). This is because load-on-init tests need to seed localStorage BEFORE the service is created. Each test calls `TestBed.inject(GameStateService)` explicitly after any required localStorage seeding.

3. **Debounce tests must call `vi.clearAllTimers()` after service injection and before mutations** to clear the pending timeout from the initial effect run (which fires on service creation). Then call `TestBed.flushEffects()` after mutations and `vi.advanceTimersByTime(500)` to trigger the debounced save.

4. **Spy on `persistence.save` BEFORE creating GameStateService.** Get the persistence instance via `TestBed.inject(StatePersistenceService)` first, apply the spy, then inject GameStateService.

## Edge Cases Addressed

| Edge case | Handling |
|-----------|----------|
| No saved state | `load()` returns null, signals keep defaults |
| Corrupted JSON in localStorage | `load()` returns null (StatePersistenceService handles parse error), defaults used |
| Valid JSON but wrong shape (e.g., a number) | Runtime type check (`typeof saved === 'object'`) rejects it, defaults used |
| Partial snapshot (missing `totalXp`) | Each field checked independently; missing fields keep defaults |
| Negative `totalXp` in saved data | Rejected by `saved.totalXp >= 0` check, default 0 used |
| Rapid mutations (spam `addXp`) | Debounce ensures at most one save per 500ms |
| `resetState()` called | Signals zeroed (triggers auto-save of defaults), localStorage cleared immediately |
| Service destroyed mid-debounce | `DestroyRef` callback clears pending timeout |
