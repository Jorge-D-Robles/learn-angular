# Plan: T-2026-325 — Create theme CSS custom property variants for dark/station/light themes

## Task

T-2026-143 wires SettingsPage theme preference to a body class (`theme-dark`, `theme-station`, `theme-light`), but no CSS actually defines what those themes look like. The `_themes.scss` file contains empty placeholder selectors. This ticket populates those selectors with CSS custom property overrides so that switching themes has a visible effect.

Acceptance criteria (from BACKLOG.md):
- `src/styles/themes/` directory with `_dark.scss`, `_station.scss`, `_light.scss`
- Each theme sets: `--color-bg-primary`, `--color-bg-secondary`, `--color-text-primary`, `--color-text-secondary`, `--color-accent`, `--color-accent-secondary`, `--color-success`, `--color-error`
- Station theme uses visual-style.md palette
- Dark theme: deeper blacks, higher contrast, same accent colors
- Light theme: white backgrounds, dark text, adapted accent colors
- `styles.scss` imports theme files and applies via body class selectors
- All existing components using hardcoded colors updated to use custom properties
- WCAG AA contrast ratios (4.5:1 for text)

## Architecture

### Decision: Override `--nx-*` tokens, not introduce a second property layer

The ticket AC specifies semantic names like `--color-bg-primary`. However, the entire codebase already consumes `--nx-*` tokens (defined in `_tokens.scss`). Introducing a parallel `--color-*` system would mean:
1. Every component needs to be rewritten to use the new names.
2. Two sets of custom properties exist simultaneously, causing confusion.
3. The one component (module-assembly) that uses `--color-text-primary` is already an outlier.

**Instead**, each theme file will override the existing `--nx-*` custom properties. The `:root` block in `_tokens.scss` defines the station theme values (they already match visual-style.md). Each `body.theme-*` selector overrides the subset of `--nx-*` properties that change per theme. This approach:
- Requires zero changes to components that already use `var(--nx-color-*)`.
- Keeps a single source of truth for property names.
- Is the standard CSS theming pattern (cascade override).

### Decision: Add missing semantic aliases to `_tokens.scss`

Several component SCSS files reference undeclared properties with inline fallbacks:
- `--nx-color-surface` (used by level-select, profile, settings, minigame-hub) -- alias for `--nx-color-hull`
- `--nx-color-border` (used by dashboard, mission) -- alias for `--nx-color-bulkhead`
- `--nx-color-surface-hover` (used by dashboard) -- slightly lighter than hull
- `--nx-color-success` (used by dashboard) -- alias for `--nx-color-sensor-green`
- `--nx-color-deep-space` (used by level-select) -- alias for `--nx-color-void`

These need to be defined in `_tokens.scss` `:root` so they exist without fallbacks, and then overridden per-theme. This is a small addition (~10 lines to `:root`).

### Decision: Scope "hardcoded color updates" to SCSS files only

The AC says "all existing components that use hardcoded colors are updated." The TypeScript files contain ~60 hardcoded hex colors, but these are almost entirely in minigame type constant maps (e.g., `PART_SLOT_COLORS`, `NODE_TYPE_COLORS`, `WIRE_TYPE_COLORS`) used for SVG rendering of game visuals. These are game-logic colors, not UI-theme colors -- a red wire should stay red regardless of light/dark mode. Changing these would require refactoring every minigame's rendering pipeline, which is a separate concern.

The SCSS files with hardcoded fallback values (the `var(--nx-color-foo, #hexval)` pattern) are already correctly using custom properties -- the fallback is just a safety net. No changes needed there.

The two actual problems are:
1. **`speed-run.ts` inline styles** -- 4 hardcoded hex colors in component `styles` array. These should use `var(--nx-*)` tokens.
2. **`terminal-hack.component.ts` computed color** -- returns hex strings for `powerGaugeColor`. This is a programmatic SVG color, similar to the minigame constants -- leave as-is since it's a gauge rendered in SVG, not a themed UI element.
3. **`module-assembly.component.scss`** -- uses `--color-*` (unprefixed) properties. Should be migrated to `--nx-color-*`.

### Theme color definitions

**Station theme** (default, matches visual-style.md exactly):
- Background: Void `#0A0E1A`, Hull `#1A1F2E`
- Text: Display `#E2E8F0`, Corridor `#8B92A8`
- Accents: Reactor Blue `#3B82F6`, Sensor Green `#22C55E`
- Error: Emergency Red `#EF4444`

**Dark theme** (deeper blacks, higher contrast):
- Background: `#050810` (deeper than Void), `#0F1320` (deeper than Hull)
- Text: `#F1F5F9` (brighter than Display), `#94A3B8` (brighter than Corridor)
- Accents: same Reactor Blue, Sensor Green (already high-contrast on dark)
- Borders: `#1A1F2E` (darker bulkhead)

**Light theme** (inverted for readability):
- Background: `#FFFFFF`, `#F8FAFC`
- Text: `#0F172A` (dark slate), `#475569` (medium slate)
- Heading: `#0F172A` (was Beacon white, now dark)
- Accents: `#2563EB` (darker blue for AA on white), `#16A34A` (darker green for AA)
- Error: `#DC2626` (darker red for AA)
- Code bg: `#F1F5F9` (light gray)
- Links/focus: adjusted blue

WCAG AA verification (4.5:1 minimum for normal text):
- Light: `#0F172A` on `#FFFFFF` = 17.4:1 (pass)
- Light: `#475569` on `#FFFFFF` = 7.1:1 (pass)
- Light: `#2563EB` on `#FFFFFF` = 4.6:1 (pass, just meets AA)
- Dark: `#F1F5F9` on `#050810` = 18.9:1 (pass)
- Dark: `#94A3B8` on `#050810` = 8.2:1 (pass)
- Station: `#E2E8F0` on `#0A0E1A` = 15.2:1 (pass)
- Station: `#8B92A8` on `#0A0E1A` = 7.5:1 (pass)

## Implementation Order

1. **Add semantic aliases to `_tokens.scss`** -- Add `--nx-color-surface`, `--nx-color-border`, `--nx-color-surface-hover`, `--nx-color-success`, `--nx-color-deep-space` to the `:root` block, mapping them to existing SCSS variables. This ensures components that already reference these properties no longer need fallbacks. (~10 LOC)

2. **Create `src/styles/themes/_station.scss`** -- Define a SCSS map or mixin containing station theme values for all theme-sensitive properties. The station theme values match the existing `:root` defaults, so this file primarily documents the station palette and provides explicit overrides for the semantic aliases. (~30 LOC)

3. **Create `src/styles/themes/_dark.scss`** -- Override `--nx-color-void`, `--nx-color-hull`, `--nx-color-bulkhead`, `--nx-color-corridor`, `--nx-color-display`, `--nx-color-beacon`, `--nx-color-code-bg`, plus the semantic aliases. Keep accent colors identical to station. (~30 LOC)

4. **Create `src/styles/themes/_light.scss`** -- Override all base colors to light equivalents, adjust accent colors for WCAG AA on white backgrounds, invert code block background. This is the most divergent theme. (~35 LOC)

5. **Update `_themes.scss`** -- Replace the empty `body.theme-dark {}` etc. selectors. Each selector block sets its theme's custom property overrides. Import the theme files using `@use`. (~25 LOC)

6. **Update `speed-run.ts` inline styles** -- Replace `#22c55e` with `var(--nx-color-sensor-green)`, `#f59e0b` with `var(--nx-color-alert-orange)`, `#ef4444` with `var(--nx-color-emergency-red)`. (~4 LOC changed)

7. **Update `module-assembly.component.scss`** -- Replace `--color-text-secondary` with `--nx-color-corridor`, `--color-text-primary` with `--nx-color-display`, `--color-accent` with `--nx-color-alert-orange`. Remove the inline fallback hex values since the properties are now always defined. (~9 LOC changed)

8. **Run full test suite** -- Verify no regressions.
