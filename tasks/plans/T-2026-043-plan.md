### Section 1: Angular App Structure

- **Tech stack**: Angular 21, TypeScript, Vitest, Playwright, SCSS, standalone components (no NgModules)
- **Directory layout** (tree diagram):
  ```
  src/app/
    core/           -- Singleton services, state, engine framework
      curriculum/   -- StoryMission types and CURRICULUM constant
      error/        -- GlobalErrorHandler
      levels/       -- LevelDefinition, LevelPack, LevelLoaderService, LevelProgressionService, LevelNavigationService
      integration/  -- Cross-service integration specs
      minigame/     -- MinigameEngine, types, registry, shell, scoring, hints, combos, drag-drop, keyboard, endless-mode, speed-run
      notifications/-- XpNotificationService, RankUpNotificationService
      persistence/  -- StatePersistenceService
      progression/  -- XpService, MasteryService, SpacedRepetitionService, StreakService, GameProgressionService, DailyChallengeService, RefresherChallengeService, etc.
      settings/     -- SettingsService
      state/        -- GameStateService, rank constants
    data/           -- Static level/mission data files
      levels/       -- Per-minigame level packs (e.g., module-assembly.data.ts)
      missions/     -- Phase-organized story mission data (phases 1-6)
      tutorials/    -- Tutorial content (scaffolded, not yet populated)
    features/       -- Feature modules for each minigame (scaffolded barrel exports)
      minigames/    -- 12 subdirectories, one per minigame
    pages/          -- Route-level page components (11 pages)
    shared/         -- Reusable UI components, directives, pipes, icons
      components/   -- 20 shared components (nx- prefix)
      directives/   -- DraggableDirective, DropZoneDirective
      pipes/        -- TimeFormatPipe
      icons/        -- Lucide icon configuration
    bottom-nav/     -- Mobile bottom navigation
    side-nav/       -- Desktop side navigation
  ```
- **Routing**: Flat route table in `app.routes.ts`, lazy-loaded via `loadComponent()` (no route modules), 11 routes (10 named routes plus a `**` wildcard 404 catch-all) including parameterized minigame routes (`/minigames/:gameId/level/:levelId`)
- **Selector conventions**: `app-` prefix for app-level components, `nx-` prefix for shared library components
- **File naming**: Angular 2025 convention -- `component-name.ts` not `component-name.component.ts`
- **Barrel exports**: Hierarchical barrel pattern -- each `core/` subdirectory has `index.ts`, `core/index.ts` re-exports all, `shared/index.ts` re-exports components/directives/pipes/icons

### Section 2: Minigame Framework Architecture

- **MinigameEngine<TLevelData>**: Abstract base class (not an Angular service -- instantiated per session via factory functions). Documents:
  - Generic type parameter for game-specific level data
  - Signal-based state: `score`, `lives`, `status`, `timeRemaining`, `currentLevel`, `playMode` (all read-only signals)
  - Lifecycle state machine: `Loading -> Playing -> Paused/Won/Lost` (include ASCII state diagram)
  - Abstract methods subclasses must implement: `onLevelLoad()`, `onStart()`, `onComplete()`, `validateAction()`
  - Optional hooks: `onPause()`, `onResume()`
  - Built-in timer with 1-second interval
  - Auto-pause on page visibility change (`_autoPaused` flag pattern)
  - Optional ComboTrackerService integration
  - `submitAction()` pipeline: validate -> update score -> check lives -> auto-fail
- **MinigameShellComponent**: Container component with content projection (`<ng-content />`). Documents:
  - Signal inputs for all HUD data (score, lives, timer, hints, status)
  - Embedded sub-components: PauseMenuComponent, LevelResultsComponent, LevelFailedComponent
  - Timer color transitions (green -> orange -> red) with configurable thresholds
  - Output events for game actions (pause, resume, retry, quit, nextLevel, requestHint)
- **MinigameRegistryService**: Map-backed registry. Documents:
  - Pre-registers all 12 minigame configs at construction
  - `register()` for adding component types and engine factories
  - `getComponent()` returns `null` (registered, no component) vs `undefined` (not registered)
  - `getEngineFactory()` for per-game engine instantiation
  - DEFAULT_MINIGAME_CONFIGS constant with all 12 configs including scoreConfig
- **MinigamePlayPage**: Route-level orchestrator. Documents:
  - 5 view states: not-found, not-ready, locked, error, ready
  - Engine lifecycle via `effect()`: create engine -> load level -> initialize -> set play mode -> start
  - Completion detection via separate `effect()` watching `engine.status()`
  - LevelCompletionService integration for XP/progression pipeline
  - NgComponentOutlet for dynamic minigame component rendering
- **Supporting services**: ComboTrackerService, HintService, KeyboardShortcutService, ScoreCalculationService, DragDropService, WireDrawService, EndlessModeService, SpeedRunService

### Section 3: State Management

- **Pattern**: Angular signals (not NgRx, not RxJS BehaviorSubjects). Document the 4-part pattern from GameStateService:
  1. Private mutable signals (`signal()`)
  2. Public read-only signals (`.asReadonly()`)
  3. Computed signals (`computed()`) for derived values
  4. Named mutation methods with validation
- **GameStateService**: Root state holder for playerName and totalXp. Rank derived via `computed()` from rank thresholds.
- **Service graph**: Show which services depend on which (facade pattern):
  - XpService facades GameStateService + StreakService
  - LevelCompletionService facades LevelProgressionService + XpService + MasteryService + XpDiminishingReturnsService + HintService
  - LifetimeStatsService facades 6 services into a single ProfileStats computed signal
- **No global store**: Each domain service owns its own signal state. No centralized store pattern.

### Section 4: Level/Content Data Format

- **LevelDefinition<T>**: Generic interface for content authoring. Documents all fields: `levelId`, `gameId`, `tier`, `order`, `title`, `conceptIntroduced`, `description`, `parTime?`, `data: T`.
- **LevelPack**: Collection of LevelDefinition<unknown> for a single minigame.
- **MinigameLevel<T>**: Lighter runtime type used by MinigameEngine. Subset of LevelDefinition fields.
- **Data file convention**: Static TypeScript arrays in `src/app/data/levels/`. Example: `module-assembly.data.ts` exports `MODULE_ASSEMBLY_LEVELS` (18 levels across 4 tiers) plus game-specific types (BlueprintSlotType, ComponentPart, BlueprintSlot, etc.).
- **Tier structure**: 4 tiers (Basic, Intermediate, Advanced, Boss) with ascending XP rewards (15/20/30/150).
- **Registration flow**: LevelLoaderService.registerLevelPack() -> LevelProgressionService.registerLevels() at app startup.
- **Story missions**: `CURRICULUM` constant in `core/curriculum/curriculum.data.ts` is typed `readonly CurriculumPhase[]` — a phase-grouped structure. Each `CurriculumPhase` contains `StoryMission` chapters. A derived flat array `ALL_STORY_MISSIONS` is also exported. The implementer must count the actual mission objects before writing the number.

### Section 5: Code Editor Integration

- **Decision**: Custom lightweight syntax highlighter (no third-party editor library). Rationale: minigames need read-only code display with line highlighting, not a full IDE.
- **CodeEditorComponent** (`nx-code-editor`): Dual-layer architecture:
  - Pre/code layer for syntax-highlighted display
  - Hidden textarea layer for editable mode (synced scrolling)
- **Syntax highlighting**: Custom regex tokenizer in `syntax-highlight.ts`. Supports TypeScript and HTML. Token types: keyword, string, comment, decorator, number, tag, attr, punctuation, text. Applied via CSS classes (`token-keyword`, `token-string`, etc.).
- **API**: Signal inputs for `code`, `language`, `readOnly`, `highlightLines`. Output: `codeChange`.
- **ExpressionBuilderComponent**: Guided mode (token selection) + raw mode (text input) for building Angular expressions. Uses its own expression-validator for token-based validation.

### Section 6: Progression Persistence

- **StatePersistenceService**: Central localStorage abstraction. Documents:
  - Namespaced keys: `nexus-station:{key}` prefix to avoid collisions
  - `save(key, data)` / `load<T>(key)` / `clear(key)` / `clearAll()`
  - `autoSave(key, source, injector?)`: Creates an `effect()` that watches a signal and persists changes
  - `exportState()` / `importState(json)`: Full state backup/restore as JSON string
- **Persistence patterns** (two distinct approaches):
  1. **`autoSave()` API**: `StatePersistenceService.autoSave(key, source)` creates an `effect()` that watches a signal and persists on change. Does NOT debounce internally.
  2. **Manual debounced-effect pattern**: Most stateful services implement their own persistence via `effect()` -> `untracked()` -> `_debouncedSave()` -> `setTimeout(500ms)` -> `DestroyRef.onDestroy()` clears pending timeout. This is the more common pattern.
  3. **Synchronous saves**: Some services (e.g., StreakRewardService, DailyChallengeService) save synchronously without debounce.
- **Services that persist** (each with its own localStorage key — implementer must audit all `persistence.save(` calls to get the complete list):
  - `game-state` -- GameStateService (playerName, totalXp)
  - `mastery` -- MasteryService (Map<MinigameId, stars>)
  - `spaced-repetition` -- SpacedRepetitionService (Map<MinigameId, epoch>)
  - `game-progression` -- GameProgressionService (Set<ChapterId>)
  - `settings` -- SettingsService (UserSettings)
  - `streak` -- StreakService
  - `streak-rewards` -- StreakRewardService
  - `diminishing-returns` -- XpDiminishingReturnsService
  - `play-time` -- PlayTimeService
  - `daily-challenge` -- DailyChallengeService
  - `endless-high-score:{gameId}` -- EndlessModeService
  - `speed-run-best-time:{gameId}` -- SpeedRunService
- **Data validation on load**: Every service validates loaded data shape/types and silently discards corrupted entries.
- **Reset**: Individual services have `reset*()` methods. SettingsService.resetProgress() calls `clearAll()` + page reload.

### Section 7: Testing Strategy

- **Framework**: Vitest (via `@angular/build:unit-test` builder), not Karma/Jasmine
- **Coverage thresholds** (enforced in CI): statements 80%, branches 75%, functions 80%, lines 80%
- **CI pipeline**: GitHub Actions with 3 parallel jobs: Lint (`ng lint`), Test (`ng test -c coverage --watch=false`), Build (`ng build`). Node 22 LTS.
- **E2E**: Playwright for smoke tests (6 tests across 2 files: `e2e/app.spec.ts` with 2 tests for dashboard routing and 404 handling, `e2e/minigame-shell.spec.ts` with 4 tests). Workaround: uses `--configuration production` due to Angular 21 Vite dev server routing bug.
- **Test helpers** (`src/testing/test-utils.ts`):
  - `createComponent<T>()`: Shorthand for TestBed configure + create + detectChanges
  - `getMockProvider<T>()`: Creates mock provider with partial overrides
- **Unit test patterns**:
  - Services: Direct instantiation via `TestBed.inject()`, mock dependencies via `getMockProvider()`
  - Components: `createComponent()` helper, query DOM for assertions
  - State services: Use `fakeAsync/tick/flush` for debounced auto-save verification
  - MinigameEngine subclasses: Extend with concrete implementations for testing abstract class behavior
- **TDD approach**: Red-Green-Refactor. Tests written before implementation. Each ticket includes specific test counts in completion summary.
- **Total**: ~83 spec files, 700+ unit tests
