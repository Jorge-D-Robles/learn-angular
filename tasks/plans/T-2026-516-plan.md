### Conclusion

**4 of 6 ACs have substantial overlap** with existing tests. The genuine gaps are:

1. `evaluateFormElements()` round-trip with real level 1 data (initialize -> place -> evaluate -> check per-element results)
2. `evaluateFormElements()` for missing/wrong elements with real level 1 data
3. `runTestCases()` per-case result verification with real data (existing test only checks aggregates)
4. Reset + re-initialize cycle with real data verifying evaluation service state is clean

The new file will contain **6 focused tests** that fill these gaps without duplicating the 8 tests in `terminal-hack.integration.spec.ts` or the engine spec's 8 synthetic-data tests. Each test is specifically about the coordinated lifecycle between the engine and the real evaluation service using real level 1 data -- something neither existing file covers end-to-end.

## Architecture

### File placement
`src/app/features/minigames/terminal-hack/evaluation-engine.integration.spec.ts` -- a new spec file in the same directory as the other integration specs.

### Pattern
Follows the same pattern as `terminal-hack.integration.spec.ts`:
- Import real `TerminalHackEngine` and `TerminalHackFormEvaluationService`
- Import `TERMINAL_HACK_LEVELS[0]` (level 1 data)
- Use the `toMinigameLevel()` helper to convert `LevelDefinition` to `MinigameLevel`
- Create engine wired to real service, initialize with real level, then exercise the coordinated lifecycle

### Key design decisions
1. **No duplication of aggregate assertions.** The existing integration spec already verifies `allPassed`, `passCount`, `status() === Won`, `score() > 0`, and 3-failure lose condition. This spec focuses on the per-element and per-case granularity that only exists when the engine delegates to the real service.
2. **Real data only.** Unlike engine spec section 15-16 which uses synthetic `makeLevel()` data, this spec uses `TERMINAL_HACK_LEVELS[0]` exclusively.
3. **6 tests, not the full 6 ACs.** ACs 4 and 5 (completion scoring, 3-failure loss) are already fully covered by `terminal-hack.integration.spec.ts` and would be pure duplication. The plan includes them only where they add new coverage (per-case detail).

## Test Plan

### Test 1: `evaluateFormElements() returns per-element results after initialize with real data`
- Create engine with real service, initialize with level 1, start
- Place correct element: `{ elementId: 'name', elementType: 'text', toolType: 'ngModel' }`
- Call `engine.evaluateFormElements()`
- Assert: returns array of length 1
- Assert: `results[0].elementId === 'name'`
- Assert: `results[0].correctType === true`
- Assert: `results[0].correctTool === true`
- Assert: `results[0].correctValidations === true` (level 1 has no validations on the spec element)
- Assert: `results[0].missingValidations` is empty array
- **Why new:** Engine spec section 15 tests this with synthetic data only. `form-evaluation.integration.spec.ts` tests `service.evaluateForm()` directly. This is the first test that goes engine -> evaluateFormElements() -> service.evaluateForm() with real data.

### Test 2: `evaluateFormElements() returns all-false for unplaced elements with real data`
- Create engine with real service, initialize with level 1, start
- Do NOT place any elements
- Call `engine.evaluateFormElements()`
- Assert: returns array of length 1
- Assert: `results[0].correctType === false`
- Assert: `results[0].correctTool === false`
- Assert: `results[0].correctValidations === false`
- **Why new:** Engine spec tests this with synthetic data that has `required` validation. Level 1 has NO validations on the spec element, so `missingValidations` should be `[]` (empty). This catches a real data shape difference.

### Test 3: `evaluateFormElements() detects wrong element type with real data`
- Create engine with real service, initialize with level 1, start
- Place element with wrong type: `{ elementId: 'name', elementType: 'email', toolType: 'ngModel' }`
- Call `engine.evaluateFormElements()`
- Assert: `results[0].correctType === false`
- Assert: `results[0].correctTool === true` (ngModel is valid for template-driven)
- **Why new:** Neither existing integration test exercises `evaluateFormElements()` with a wrong-type element against real data.

### Test 4: `runTestCases() returns per-case results through evaluation service with real data`
- Create engine with real service, initialize with level 1, start
- Place correct element
- Call `engine.runTestCases()`
- Assert: `result.testCaseResults.length === 2` (level 1 has 2 test cases)
- Assert: `result.testCaseResults[0].testCaseId === 'tc1'`
- Assert: `result.testCaseResults[0].passed === true`
- Assert: `result.testCaseResults[1].testCaseId === 'tc2'`
- Assert: `result.testCaseResults[1].passed === true`
- **Why new:** `terminal-hack.integration.spec.ts` test 1 checks aggregates (`allPassed`, `passCount`) but never inspects `testCaseResults[i]`. `form-evaluation.integration.spec.ts` calls `service.runTestCases()` directly. This is the first test that verifies per-case results through the engine's `runTestCases()` with real data.

### Test 5: `reset via re-initialize clears evaluation service state with real data`
- **Important:** Construct `service` and `engine` inline (NOT via helper) so the `service` variable is directly accessible for `getTestPassRate()`.
- Create `service = new TerminalHackFormEvaluationService()`, `engine = new TerminalHackEngine(service, { initialLives: 10 })` — `initialLives: 10` prevents `Lost` status on first failure.
- Initialize with level 1, start
- Place correct element, run tests — all pass, `service.getTestPassRate() === 1.0`
- Assert: `service.getTestPassRate() === 1.0` (non-zero, proves cache holds state)
- Re-initialize engine with level 1 (triggers `onLevelLoad` -> `service.reset()`)
- Assert: `service.getTestPassRate() === 0` (reset cleared the cached result)
- Start again, place correct element, call `evaluateFormElements()`
- Assert: returns correct per-element results (service is functional after reset)
- **Why new:** Engine spec section 16 "reset propagates to service" uses synthetic data and only checks `getTestPassRate()`. This test uses a non-zero pre-reset pass rate (1.0) to unambiguously verify the reset, then confirms the service is fully functional after reset.

### Test 6: `evaluateFormElements() and runTestCases() produce consistent results with real data`
- Create engine with real service, initialize with level 1, start
- Place correct element
- Call `engine.evaluateFormElements()` -- per-element check says "all correct"
- Call `engine.runTestCases()` -- test cases say "all pass"
- Assert: both agree (evaluateFormElements says correct, runTestCases says allPassed)
- This tests the coordinated consistency: the two service methods, accessed through two different engine methods, produce aligned results for the same placed elements.
- **Why new:** No existing test verifies that `evaluateFormElements()` and `runTestCases()` produce consistent results when called in sequence on the same engine state.
