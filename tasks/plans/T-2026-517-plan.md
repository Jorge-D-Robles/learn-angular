# Plan: T-2026-517 — Create Terminal Hack visual state integration test for UI-to-engine signal binding

## Task

Create an integration test that verifies DOM rendering **inside child components** correctly reflects engine state. The existing `terminal-hack.component.spec.ts` tests parent-level signal bindings (via `By.directive` and `componentInstance`), but does NOT query child component DOM internals. This file bridges that gap.

## Architecture

### What this test covers that existing specs do NOT

The component spec tests:
- Parent-level wiring: child component exists, input values match engine signals
- Parent-level DOM: completion %, complete badge, power gauge color transitions

The child component specs (code-panel, live-preview, test-runner) test:
- Each child in isolation via TestHost with synthetic data

**This integration spec uniquely tests:** Engine → parent → child component → rendered DOM. It queries CSS selectors INSIDE child components (`.code-panel__spec-item`, `.live-preview__slot--correct`, `.test-runner__case--pass`) rendered through the real parent with a real engine. This catches wiring bugs where the parent passes the wrong signal or the child template has a conditional that doesn't fire correctly in context.

### Scope: 5 focused tests, ~100 LOC

Tests are narrowly scoped to child component DOM assertions that no other spec file covers.

## Test Plan

### Test 1: Code panel renders target spec element labels from engine level data
- Setup with engine + level data (use `makeLevel` helper with 2 elements: 'Username' with `required` validation, 'Email' without)
- After async setup, query `.code-panel__spec-item` elements inside the rendered parent
- Assert: 2 spec items rendered
- Assert: first spec item's `.code-panel__spec-label` contains "Username"
- Assert: first spec item has `.code-panel__spec-item--required` class (has required validation)
- Assert: second spec item does NOT have `.code-panel__spec-item--required` class
- **Why new:** Component spec checks `componentInstance.targetSpec()` but never queries `.code-panel__spec-item` DOM nodes or `.code-panel__spec-item--required` class through the parent.

### Test 2: Live preview shows correct/missing slot states after element placement
- Setup with engine (2 elements in spec), start playing
- Before placement: query `.live-preview__slot` elements, all have `.live-preview__slot--missing` class
- Call `component.onPlaceElement('el-1', 'text', 'FormControl')`, `fixture.detectChanges()`
- After placement: first slot has `.live-preview__slot--correct` class (`elementType: 'text'` matches spec element type `'text'`)
- Second slot still has `.live-preview__slot--missing`
- **Why new:** Component spec checks `componentInstance.formElements()` length but never queries `.live-preview__slot--missing`/`--correct` CSS classes through the parent.

### Test 3a: Test runner shows pass indicators after successful test run
- Setup with engine, place all correct elements, call `component.onRunTests()`, `fixture.detectChanges()`
- Query `.test-runner__case--pass` inside the rendered parent
- Assert: count matches expected test case count (all pass)
- Assert: no `.test-runner__case--fail` elements present
- **Note:** Do NOT use `await fixture.whenStable()` after `onRunTests()`. The engine result is synchronous; the `setTimeout` in `onRunTests` is irrelevant to these assertions.
- **Why new:** Component spec checks `componentInstance.testResults()` but never queries `.test-runner__case--pass`/`--fail` DOM elements through the parent.

### Test 3b: Test runner shows fail indicators after failed test run (separate `it()` with fresh `setup()`)
- Setup with engine, place WRONG element (wrong elementType), call `component.onRunTests()`, `fixture.detectChanges()`
- Query `.test-runner__case--fail` inside the rendered parent
- Assert: at least 1 `.test-runner__case--fail` present

### Test 4: Power gauge fill width reflects timer state
- Call `vi.useFakeTimers()` BEFORE `setup()` — ensures `setInterval` is under fake time
- `afterEach(() => vi.useRealTimers())`
- Setup with `timeLimit: 100`, start playing
- Advance 50s via `vi.advanceTimersByTime(50_000)`, `fixture.detectChanges()`
- Query `.terminal-hack__power-gauge-fill`, assert `style.width` is `'50%'`
- **Why new:** Component spec tests gauge COLOR transitions (orange at 40%, red at 25%) but NOT the exact width% value. This test verifies the `powerGaugeRatio` signal propagates to `style.width` correctly at an arbitrary time point.
