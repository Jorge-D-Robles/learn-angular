# Plan: T-2026-017 -- Create abstract MinigameEngine base class

## Task

Create the abstract base class that all minigame engines extend. This provides the lifecycle hooks, state management, and scoring logic that every minigame shares.

## Architecture

### Design Decision: Pure TypeScript abstract class (not an Angular service)

The `MinigameEngine<TLevelData>` is a **plain abstract TypeScript class**, not an `@Injectable` Angular service. This is deliberate:

- Each minigame will create its own concrete subclass (e.g., `ModuleAssemblyEngine extends MinigameEngine<ModuleAssemblyData>`)
- Engines are instantiated per-game-session, not shared singletons
- Angular's `signal()` and `computed()` work outside of DI -- they are standalone reactive primitives
- The concrete subclass can be provided via DI if needed, but the base class itself has no Angular decorator

### State Management

State is managed via private writable signals with public read-only projections:

```
Private signals:              Public read-only:
_score (WritableSignal)   --> score (Signal)
_lives (WritableSignal)   --> lives (Signal)
_status (WritableSignal)  --> status (Signal)
_timeRemaining (Signal)   --> timeRemaining (Signal)
_currentLevel (Signal)    --> currentLevel (Signal)
```

This matches the `GameStateService` pattern (private `signal()`, public `.asReadonly()`).

### Lifecycle State Machine

```
                  initialize(level)
  [uninitialized] ----------------> [Loading]
                                       |
                                   start()
                                       |
                                       v
                   pause()         [Playing] ----> fail() ----> [Lost]
                   <------>                  ----> complete() -> [Won]
                   resume()        [Paused]
```

Valid transitions enforced by guard checks:
- `initialize()`: only from any status (resets everything)
- `start()`: only from `Loading`
- `pause()`: only from `Playing`
- `resume()`: only from `Paused`
- `complete()`: only from `Playing`
- `fail()`: only from `Playing`

Invalid transitions are silently ignored (no throw) to match game UI patterns where buttons may fire during transitions.

### Timer Management

Timer is optional. If a level has a `timerDuration` (passed via config to `initialize`), a countdown starts on `start()` and pauses/resumes with the game. When the timer reaches 0, `fail()` is automatically called.

Timer uses `setInterval` with 1-second ticks. The interval ID is stored for cleanup. The `destroy()` method clears any active timer.

### Action Submission

`submitAction(action: unknown)` calls the abstract `validateAction(action)` which returns an `ActionResult`:

```ts
interface ActionResult {
  valid: boolean;
  scoreChange: number;
  livesChange: number;  // negative values reduce lives
}
```

After validation, `submitAction`:
1. Updates score (clamped to >= 0)
2. Updates lives (if livesChange is negative)
3. Auto-calls `fail()` if lives reach 0

### Why this approach over alternatives

- **Why not an Angular service?** Engines are per-session objects, not app-wide singletons. Using DI would require factory patterns that add complexity without benefit.
- **Why signals instead of RxJS?** Signals are the Angular 21 idiomatic approach for synchronous state. The UI will bind directly to these signals. RxJS would be overkill for simple state containers.
- **Why an abstract class instead of an interface + mixin?** Abstract classes provide both the contract (abstract methods) and shared implementation (lifecycle, timer, scoring). Interfaces cannot carry implementation.
- **Why silent no-op on invalid transitions?** Game UIs often have race conditions (e.g., user clicks "pause" while a fail animation is playing). Throwing would crash the app.

#### Default config constant (exported)

```ts
export const DEFAULT_ENGINE_CONFIG: MinigameEngineConfig = {
  initialLives: 3,
  timerDuration: null,
};
```

#### Abstract class `MinigameEngine<TLevelData>`

**Private writable signals:**
- `_score = signal(0)`
- `_lives = signal(3)`
- `_status = signal(MinigameStatus.Loading)`
- `_timeRemaining = signal(0)`
- `_currentLevel = signal<string | null>(null)`

**Public read-only signals:**
- `score = this._score.asReadonly()`
- `lives = this._lives.asReadonly()`
- `status = this._status.asReadonly()`
- `timeRemaining = this._timeRemaining.asReadonly()`
- `currentLevel = this._currentLevel.asReadonly()`

**Computed signal:**
- `state = computed<MinigameState>(() => ({ ... }))` -- aggregates all signals into a `MinigameState` object

**Private fields:**
- `_config: MinigameEngineConfig`
- `_timerId: ReturnType<typeof setInterval> | null = null`

**Constructor:**
- `protected constructor(config: Partial<MinigameEngineConfig> = {})`
- Merges with `DEFAULT_ENGINE_CONFIG`

**Lifecycle methods (public):**

1. `initialize(level: MinigameLevel<TLevelData>): void`
   - Clears any existing timer
   - Resets: score=0, lives=config.initialLives, status=Loading
   - Sets `_currentLevel` to `level.id`
   - Sets `_timeRemaining` to `config.timerDuration ?? 0`
   - Calls abstract `onLevelLoad(level.data)`

2. `start(): void`
   - Guard: only if status is `Loading`
   - Sets status to `Playing`
   - If timer configured, starts the countdown interval
   - Calls abstract `onStart()`

3. `pause(): void`
   - Guard: only if status is `Playing`
   - Sets status to `Paused`
   - Clears timer interval (preserving remaining time)

4. `resume(): void`
   - Guard: only if status is `Paused`
   - Sets status to `Playing`
   - Restarts timer interval if timer was configured

5. `complete(): void`
   - Guard: only if status is `Playing`
   - Clears timer
   - Sets status to `Won`
   - Calls abstract `onComplete()`

6. `fail(): void`
   - Guard: only if status is `Playing`
   - Clears timer
   - Sets status to `Lost`

7. `destroy(): void`
   - Clears any active timer interval
   - Resets score=0, lives=0, timeRemaining=0, currentLevel=null
   - Sets status to `Lost` (NOT `Loading`) — prevents accidental restart after destroy. The engine is not reusable after `destroy()`; create a new instance instead.

**Action method (public):**

8. `submitAction(action: unknown): ActionResult`
   - Guard: only if status is `Playing`, otherwise returns `{ valid: false, scoreChange: 0, livesChange: 0 }`
   - Calls abstract `validateAction(action)` to get `ActionResult`
   - Updates `_score`: `Math.max(0, current + result.scoreChange)`
   - Updates `_lives`: `current + result.livesChange` (applies both positive and negative changes — positive allows power-ups/life restoration)
   - If lives <= 0 after update, calls `fail()`
   - Returns the `ActionResult`

**Private methods:**

9. `_startTimer(): void`
   - Sets up `setInterval` at 1000ms
   - Each tick decrements `_timeRemaining` by 1
   - When `_timeRemaining` reaches 0, calls `fail()`

10. `_clearTimer(): void`
    - Clears interval if `_timerId` is not null
    - Sets `_timerId` to null

**Abstract methods (protected):**
- `abstract onLevelLoad(data: TLevelData): void`
- `abstract onStart(): void`
- `abstract onComplete(): void`
- `abstract validateAction(action: unknown): ActionResult`

### `minigame-engine.spec.ts`

Create a concrete `TestEngine` subclass for testing:

```ts
class TestEngine extends MinigameEngine<{ difficulty: number }> {
  onLevelLoadCalled = false;
  onStartCalled = false;
  onCompleteCalled = false;
  lastValidatedAction: unknown = null;
  nextValidationResult: ActionResult = { valid: true, scoreChange: 10, livesChange: 0 };

  constructor(config?: Partial<MinigameEngineConfig>) {
    super(config);
  }

  protected onLevelLoad(data: { difficulty: number }): void {
    this.onLevelLoadCalled = true;
  }
  protected onStart(): void { this.onStartCalled = true; }
  protected onComplete(): void { this.onCompleteCalled = true; }
  protected validateAction(action: unknown): ActionResult {
    this.lastValidatedAction = action;
    return this.nextValidationResult;
  }
}
```

**Test suites and cases:**

1. **Initialization**
   - should default to Loading status before initialize is called
   - should set status to Loading after initialize
   - should set currentLevel to the level id
   - should reset score to 0
   - should set lives to config initialLives (default 3)
   - should set timeRemaining to timerDuration when configured
   - should set timeRemaining to 0 when no timer configured
   - should call onLevelLoad with level data

2. **Lifecycle: initialize -> start -> complete (happy path)**
   - should set status to Playing after start
   - should call onStart
   - should set status to Won after complete
   - should call onComplete

3. **Lifecycle: initialize -> start -> fail**
   - should set status to Lost after fail
   - should clear the timer on fail

4. **Lifecycle: pause and resume**
   - should set status to Paused after pause
   - should set status to Playing after resume
   - should not allow pause from non-Playing status
   - should not allow resume from non-Paused status

5. **Invalid transitions (no-op guards)**
   - should not allow start from Playing
   - should not allow start from Won
   - should not allow complete from Paused
   - should not allow complete from Loading
   - should not allow fail from Loading
   - should not allow fail from Won

6. **Re-initialization**
   - should allow initialize to reset from any state (Won/Lost)
   - should clear timer when re-initializing

7. **submitAction**
   - should call validateAction with the action
   - should update score based on scoreChange
   - should not allow score to go below 0
   - should update lives based on livesChange
   - should auto-fail when lives reach 0
   - should return the ActionResult
   - should no-op when status is not Playing

8. **Timer management** (use `import { vi } from 'vitest'` and `vi.useFakeTimers()`)
   - should start countdown on start when timerDuration is set
   - should decrement timeRemaining each second
   - should auto-fail when timer reaches 0
   - should pause timer on pause
   - should resume timer on resume (continues from remaining whole-second count, not fractional seconds)
   - should not start timer when timerDuration is null

9. **Read-only signal enforcement**
   - should throw when attempting to set score directly
   - should throw when attempting to set status directly

10. **destroy()**
    - should clear timer on destroy
    - should set status to Lost (not reusable after destroy)
    - should reset score to 0, lives to 0, currentLevel to null
    - should not allow start() after destroy (Lost status blocks it)

11. **Computed state signal**
    - should aggregate all signals into a MinigameState object
    - should update reactively when individual signals change
