# Plan: T-2026-318 — Create MinigamePlayPage retry flow integration test

## Task
T-2026-309 fixes HintService reset on retry, and T-2026-184 integrates ComboTrackerService with MinigameEngine. The retry flow involves resetting multiple services (hints, combo, engine state, completion flag). No integration test verifies the full retry pipeline: player fails/completes -> clicks retry -> all services reset -> new attempt starts clean. A regression in any reset step would silently corrupt scoring or progression.

Acceptance criteria:
- Integration test at `src/app/pages/minigame-play/minigame-play-retry.integration.spec.ts`
- Test: complete level with hints used -> retry -> verify hintService.hasUsedHints() is false
- Test: complete level with combo active -> retry -> verify comboTracker is reset to 0
- Test: complete level -> retry -> complete again -> verify LevelCompletionService called twice
- Test: fail level -> retry -> verify engine status returns to Playing
- Uses TestBed with real services where possible

## Architecture

This test file follows the established integration test pattern from `minigame-play.integration.spec.ts`. Key decisions:

**Real services vs mocks:**
- **Real:** HintService, LevelLoaderService, LevelProgressionService, LevelCompletionService, ComboTrackerService — these are the services whose reset behavior we are testing end-to-end.
- **Mock:** MinigameRegistryService (returns test engine/component), LevelNavigationService (returns null for next level), ActivatedRoute (returns fixed params).
- **Spy (call-through):** `LevelCompletionService.completeLevel` is spied with `vi.spyOn()`, which calls through by default in Vitest.

**TestEngine with ComboTracker:** The `TestEngine` subclass accepts an optional `ComboTrackerService` via the engine config's `comboTracker` option. The engine's `initialize()` method calls `comboTracker.reset()` — this is engine behavior that we verify is correctly wired through `onRetry()`.

**Why a separate file:** The existing `minigame-play.integration.spec.ts` covers the completion-to-progression pipeline. This file focuses exclusively on retry flow (reset correctness).

**Test 3 vs existing re-completion test:** The existing `minigame-play.integration.spec.ts` already tests that `completeLevel` is called twice after retry. Our Test 3 adds value by verifying the intermediate state: `completionSummary()` is `null` between retry and second completion, which the existing test does not check.

**HintService.reset() side effect:** `hintService.reset()` clears both `_usedHints` and `_config` (resets `maxScore` to 0). `hintService.configure()` is called once during setup and is NOT re-called after retry. Tests must not assert penalty values after retry since `_config.maxScore` is 0. Hint registrations survive reset — `getRemainingHints()` returns the original count.

## Test Plan

### Test 1: Hint reset on retry
**Setup:** Register 2 hints for the test level. Start engine. Request a hint (consuming one). Complete the engine.
**Action:** Call `component.onRetry()`.
**Assert:**
- `hintService.hasUsedHints()` returns `false`
- `hintService.getRemainingHints('ma-basic-01')` returns 2 (registrations survive reset)
- `component.activeHintText()` is `''` (UI hint text cleared)

### Test 2: Combo reset on retry (wiring test — verifies engine.initialize() resets combo via onRetry)
**Setup:** Create `TestEngine` with real `ComboTrackerService` injected via `comboTracker` config. After engine starts, call `testEngine.recordCorrectAction()` 5 times to build combo.
**Pre-condition assert:** `comboTracker.currentCombo()` is 5 (proves combo was built before retry).
**Action:** Complete the engine, then call `component.onRetry()`.
**Assert:** `comboTracker.currentCombo()` is 0. `comboTracker.maxCombo()` is 0 (engine's `initialize()` calls `comboTracker.reset()` during retry).

### Test 3: Intermediate state cleared between completions on retry
**Setup:** Complete the engine so `completeLevelSpy` is called once. Assert `component.completionSummary()` is not null.
**Action:** Call `component.onRetry()`.
**Intermediate assert:** `component.completionSummary()` is `null` (cleared by retry).
**Action:** Call `testEngine.complete()` again.
**Assert:** `completeLevelSpy` has been called exactly 2 times. `component.completionSummary()` is not null (populated from second completion).

### Test 4: Fail -> retry -> engine status returns to Playing
**Setup:** Start engine. Call `testEngine.fail()` so status becomes `Lost`.
**Action:** Call `component.onRetry()`.
**Assert:** `testEngine.status()` is `MinigameStatus.Playing`. `component.completionSummary()` is null. Engine score is 0. Lives reset to `initialLives`.

## Detailed Implementation

### setup() helper (authoritative version)

```typescript
async function setup(options: { withComboTracker?: boolean } = {}) {
  // 1. Configure TestBed (no component yet)
  await TestBed.configureTestingModule({
    imports: [MinigamePlayPage],
    providers: [
      provideRouter([]),
      ...ICON_PROVIDERS,
      { provide: MinigameRegistryService, useValue: mockRegistry },
      { provide: LevelNavigationService, useValue: mockLevelNav },
    ],
  }).compileComponents();

  TestBed.overrideProvider(ActivatedRoute, {
    useValue: { paramMap: of(convertToParamMap({ gameId: 'module-assembly', levelId: 'ma-basic-01' })) },
  });

  // 2. Inject real services from TestBed
  const comboTracker = options.withComboTracker
    ? TestBed.inject(ComboTrackerService)
    : undefined;
  const testEngine = new TestEngine(comboTracker ? { comboTracker } : {});

  // 3. Wire engine into registry mock (factory only called when component effect runs)
  // mockRegistry and mockLevelNav are module-scope vi.fn() objects defined before describe():
  //   mockRegistry = { getComponent: vi.fn().mockReturnValue(DummyGameComponent),
  //                    getConfig: vi.fn().mockReturnValue(undefined),
  //                    getEngineFactory: vi.fn() }
  //   mockLevelNav = { getNextLevel: vi.fn().mockReturnValue(null),
  //                    getPreviousLevel: vi.fn().mockReturnValue(null),
  //                    isNextLevelUnlocked: vi.fn().mockReturnValue(false) }
  // Copy exact shapes from minigame-play.integration.spec.ts lines 108-125.
  mockRegistry.getEngineFactory.mockReturnValue(() => testEngine);

  // 4. Register level pack and hints
  TestBed.inject(LevelLoaderService).registerLevelPack(TEST_LEVEL_PACK);
  const hintService = TestBed.inject(HintService);
  hintService.registerHints('ma-basic-01', TEST_HINTS);
  hintService.configure({ maxScore: 1000 });

  // 5. Create component — triggers engine lifecycle effect
  const fixture = TestBed.createComponent(MinigamePlayPage);
  fixture.detectChanges();
  await fixture.whenStable();

  // 6. Spy on completeLevel AFTER whenStable (spy installed before any test-driven complete() call)
  const completeLevelSpy = vi.spyOn(
    TestBed.inject(LevelCompletionService),
    'completeLevel',
  );

  return { fixture, component: fixture.componentInstance, testEngine, hintService, comboTracker, completeLevelSpy };
}
```

**Note:** The `completeLevelSpy` is installed after `whenStable()`. All `testEngine.complete()` calls happen after `setup()` returns, so the spy always captures them.
