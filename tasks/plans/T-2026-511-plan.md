## Architecture

### Wiring Pattern

Follows the established P2 wiring pattern (T-2026-472 WireProtocol, T-2026-473 FlowCommander):
1. Add child to parent's `imports` array
2. Replace inline HTML with child component tag
3. Bind parent signals to child inputs
4. Forward child outputs to parent handler

### What Gets Replaced

The entire `terminal-hack__content` div (lines 14-71 of HTML) contains:
- **Left panel (lines 16-57):** Interactive spec list + completion + hints
- **Right panel (lines 59-70):** Tool palette + read-only code editor

The code panel replaces the **spec display** (left) and **palette + editor** (right). The completion indicator and hints section stay in the parent but move to a sidebar alongside the code panel.

### Interactive Placement Workflow â€” Explicitly Dormant

The parent currently has an interactive workflow: click spec item â†’ select slot â†’ click tool button â†’ place element. The code panel is **display-only** (no click handlers for placement). After this wiring:

- The interactive placement template elements (clickable spec items, clickable tool buttons) are **removed from the parent template**
- The component methods (`onSelectSlot`, `onToolSelected`, `onPlaceElement`, `onRemoveElement`) **stay on the component class** as public methods (they are engine delegation methods that will be reconnected in T-2026-516)
- The signals `selectedSlotId` and `placedElementIds` **stay** (used by those methods)
- **Tests that exercise these methods via DOM clicks are DELETED** (the DOM surface no longer exists). Direct method-call tests (if any) are kept.
- A future ticket (T-2026-516: wire engine integration) will restore the interactive placement UI via a different mechanism

This is consistent with how P2 wiring tickets work: they wire in the child component, and some parent functionality becomes temporarily dormant until full integration.
