3. **Add the Terminal Hack: Form Evaluation Pattern subsection** (`### Terminal Hack: Form Evaluation Pattern`). Document the 3-phase pipeline:

   Phase 1 -- Element Placement: Player places form elements via `PlaceElementAction`, `RemoveElementAction`, and `SetValidationAction` through the engine's `submitAction()`. The engine maintains a `ReadonlyMap<string, PlayerFormElement>` of placed elements as a signal.

   Phase 2 -- Form Evaluation: `TerminalHackFormEvaluationService.evaluateForm()` compares placed elements against the `TargetFormSpec`, checking element type, tool type compatibility (template-driven vs reactive), and validation rules. Returns per-element `ElementEvaluationResult[]`.

   Phase 3 -- Test Case Execution: `TerminalHackFormEvaluationService.runTestCases()` runs predefined `FormTestCase[]` against placed elements. Each test case provides `inputValues` (Record<string, string>) and `expectedValid` (boolean) plus optional `expectedErrors`. The service applies validation rules to inputs and compares actual validity against expected. Returns `TestRunResult` with pass/fail per case and aggregate `passRate`.

   Include the scoring formula: `Math.max(0, Math.round(maxScore * correctnessRatio * speedMultiplier * attemptMultiplier) - hintDeduction)`. Note: `correctnessRatio` equals `testResult.passRate`. The result is rounded before subtracting the hint deduction and floored at 0. Note the 3-life lose condition (each failed test run dispatches a `TestFailureAction` with `livesChange: -1` through the engine's standard action pipeline).

4. **Add the Terminal Hack: Live Preview subsection** (`### Terminal Hack: Live Preview`). Document the `TerminalHackLivePreviewComponent` pattern:
   - Input: `TargetFormSpec` (the target) + `PlayerFormElement[]` (what the player has placed)
   - Computed `previewSlots` signal: maps each spec element to a `PreviewSlot` with status `'missing'` (no element placed for that slot) | `'correct'` (placed element's `elementType` matches spec's `elementType`) | `'incorrect'` (placed but type-mismatched)
   - Uses a `Map<string, PlayerFormElement>` lookup computed from the input array for O(1) matching
   - The engine also exposes a `formPreview` computed signal with `completionRatio` for the sidebar progress indicator

5. **Add the Terminal Hack: Test Runner subsection** (`### Terminal Hack: Test Runner`). Document the `TerminalHackTestRunnerComponent`:
   - Displays test cases with pass/fail indicators
   - Uses a `Map<string, TestCaseResult>` lookup for O(1) result matching per test case
   - Staggered animation delay (`i * 80ms`) for visual test execution effect
   - "Run Tests" button triggers `engine.runTestCases()` which either auto-completes (all pass) or costs a life (any fail)

6. **Add the Terminal Hack: Visual Style subsection** (`### Terminal Hack: Visual Style`). Document the retro terminal aesthetic:
   - Green-on-black color scheme (`#00ff41` on `#0a0a0a`)
   - CSS scanline overlay using `repeating-linear-gradient` with 8s flicker animation
   - Monospace font (`var(--nx-font-mono)`)
   - Power gauge timer (horizontal bar depleting from green -> orange -> red, replacing the shell's built-in timer)
   - `prefers-reduced-motion` support: disables scanline animation and power gauge transitions
   - This is the first game-specific visual pattern; the section notes that future minigames may introduce their own themed aesthetics

7. **Update the section header count** in line 3 of architecture.md. Currently says "Last updated after completion of 128 P1 tickets." -- update to note P4 Terminal Hack additions. Change to: "Last updated after P4 (Terminal Hack). Every claim is traceable to actual source files."
