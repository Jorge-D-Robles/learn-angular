# Plan: T-2026-089 — Register Terminal Hack in MinigameRegistry and wire routes

## Task

Register Terminal Hack minigame in MinigameRegistryService and ensure end-to-end playability. Same pattern as T-2026-084 (Corridor Runner registration).

Acceptance criteria:
- Terminal Hack registered in MinigameRegistryService
- Navigating to `/minigames/terminal-hack/level/1` loads the game
- MinigameShell integration works
- E2E smoke test: navigate to level 1, verify game renders

## Architecture

This follows the exact pattern established by T-2026-084 (Corridor Runner) and the P2 games before it. Three files are touched:

1. **`app.config.ts`** — Add `provideLevelData(TERMINAL_HACK_LEVEL_PACK)` and `provideMinigame('terminal-hack', TerminalHackComponent, () => new TerminalHackEngine())` to the providers array. The `TerminalHackEngine` constructor takes an optional `Partial<MinigameEngineConfig>` and an optional `TerminalHackSimulationService` interface — both can be omitted for basic registration (the interface has no concrete class exported from the barrel, matching the self-contained engine pattern).

2. **`provide-minigame.spec.ts`** — Add a new `describe('provideMinigame — terminal-hack')` block with 3 tests (component registration, engine factory, config preservation). No tutorial steps test because terminal-hack has no tutorial data yet (same as corridor-runner).

3. **`e2e/minigame-shell.spec.ts`** — Add a new E2E test `'should render Terminal Hack game for a valid level'` navigating to `/minigames/terminal-hack/level/th-basic-01`. Update the existing "coming-soon" test to use `power-grid` instead of `terminal-hack` (since terminal-hack will now be registered).

**Why this approach:** The `provideMinigame` + `provideLevelData` pattern is the established registration mechanism. No new patterns or abstractions are needed. The engine has no external service dependencies that need DI wiring (unlike Signal Corps with its shared `SignalCorpsWaveService` or Corridor Runner with `CorridorRunnerSimulationService`).

## Test Plan

### Unit tests (provide-minigame.spec.ts)

Add a new `describe('provideMinigame — terminal-hack')` block:

1. **`should register the component type for terminal-hack`** — Verify `registry.getComponent('terminal-hack')` returns `TerminalHackComponent`.
2. **`should register an engine factory that produces a TerminalHackEngine`** — Verify `registry.getEngineFactory('terminal-hack')` returns a factory, call it, assert `engine instanceof TerminalHackEngine`.
3. **`should preserve the existing config for terminal-hack`** — Verify `registry.getConfig('terminal-hack')` returns config with `name === 'Terminal Hack'`.

No tutorial steps test needed (no tutorial data exists for terminal-hack yet, same situation as corridor-runner at registration time).

### E2E tests (minigame-shell.spec.ts)

1. **New test: `should render Terminal Hack game for a valid level`** — Navigate to `/minigames/terminal-hack/level/th-basic-01`, assert `app-minigame-shell` and `app-terminal-hack` selectors are visible.
2. **Updated test: `should navigate back to Minigame Hub from coming-soon state`** — Change URL from `/minigames/terminal-hack/level/1` to `/minigames/power-grid/level/1` (power-grid is the next unregistered game).
