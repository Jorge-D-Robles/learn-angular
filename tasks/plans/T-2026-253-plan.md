# Plan: T-2026-253 — Wire LeaderboardComponent into LevelSelectPage replay mode tabs

## Task

LevelSelectPage (T-2026-077) specifies "Replay mode tabs" but no ticket wires LeaderboardComponent (T-2026-219) into those tabs. Gamification research says leaderboards drive competitive replay.

Acceptance criteria:
- [ ] LevelSelectPage includes a "Leaderboard" tab in the replay mode section
- [ ] Tab renders `nx-leaderboard` with the current minigame's `gameId`
- [ ] Mode tabs (story/endless/speedrun) are passed through
- [ ] Unit tests for: leaderboard tab presence, gameId binding

## Architecture

The current LevelSelectPage has a `TABS` array defining 4 mode tabs (Story, Endless, Speed Run, Daily). When a non-Story tab is active, the template renders a "mode launcher" section with a description and launch button. The ticket adds a 5th tab, "Leaderboard," that renders the `nx-leaderboard` component instead of the mode launcher.

**Key design decisions:**

1. **Add "Leaderboard" as a new PlayMode-like concept, NOT a new PlayMode enum value.** The `PlayMode` enum is used throughout the engine, scoring, and minigame state -- adding a value there would ripple across the codebase. Instead, the `TabConfig` already uses `mode: PlayMode` but we can change the tab system to use a broader union type. The cleanest approach: widen the tab's `mode` field type to `PlayMode | 'leaderboard'` (a string literal union) so the tab system can track "Leaderboard" as a pseudo-mode without polluting the engine's `PlayMode` enum.

2. **activeTab signal type widens to `PlayMode | 'leaderboard'`.** This is a local change scoped to the LevelSelectPage. All existing conditional checks (`activeTab() === PlayMode.Story`, `activeTab() !== PlayMode.Story`) remain correct because `'leaderboard'` is neither `PlayMode.Story` nor any other `PlayMode` value. We add one new template branch: `@if (activeTab() === 'leaderboard')` that renders the `nx-leaderboard` component.

3. **LeaderboardComponent mode passthrough.** The `nx-leaderboard` component has its own internal mode tabs (Story, Endless, Speed Run). The ticket says "Mode tabs (story/endless/speedrun) are passed through." Since LeaderboardComponent already has internal tabs and defaults to `'story'`, we simply render it without an explicit `[mode]` binding -- letting LeaderboardComponent manage its own tab state. The `gameId` signal from the route is passed through via `[gameId]`.

4. **Why not nest the leaderboard inside the mode launcher?** The acceptance criteria say "LevelSelectPage includes a 'Leaderboard' tab" -- this means it is a peer tab alongside Story/Endless/Speed Run/Daily, not a sub-element of one of them. The gamification research confirms leaderboards should be per-minigame (scoped by gameId), and the LeaderboardComponent already handles mode filtering internally.

5. **Hiding mode launcher when Leaderboard tab is active.** The current template shows the mode launcher when `activeTab() !== PlayMode.Story`. With the addition of the Leaderboard tab, this condition changes to: show mode launcher when the active tab is not Story AND not Leaderboard. The cleanest guard is `activeTab() !== PlayMode.Story && activeTab() !== 'leaderboard'`.

6. **Tab ordering.** The Leaderboard tab goes last (after Daily) since it is a secondary/meta feature, not a play mode.
