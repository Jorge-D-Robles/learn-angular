## Architecture

This is a pure test file -- no production code changes. The test differs from the existing `terminal-hack.integration.spec.ts` (T-2026-194) in its focus:

- **T-2026-194** (`terminal-hack.integration.spec.ts`): Tests the engine pipeline in depth using **only level 1** (th-basic-01). Covers state transitions, scoring with time/hint/attempt penalties.
- **T-2026-515** (`level-data-compat.integration.spec.ts`): Tests that **all 21 levels** are structurally compatible with the engine. Iterates every level to catch data authoring bugs. Includes one deeper test for level 1 evaluation and one for engine reset across levels.

**Why iterate all levels instead of spot-checking a few:** The 21 levels span 4 tiers with different form types (template-driven vs reactive), different available tools, and different validation complexities. A data authoring bug in any single level (e.g., a missing `id` field, an `availableElements` array with an invalid tool type) would only surface when that specific level is loaded. The `terminal-hack.data.spec.ts` already checks structural properties (counts, ordering, field presence) but does NOT test that each level can actually be loaded into the engine via `initialize()`. This test bridges that gap.

**Why no fake timers needed:** The level data compat tests focus on whether `initialize()` succeeds and whether the engine's signals contain valid data after initialization. The only test that calls `start()` + `runTestCases()` is the level 1 evaluation test, which completes fast enough that the 1-second timer interval does not fire. The reset test also completes instantly.

**toMinigameLevel helper:** Reuses the same conversion pattern as `terminal-hack.integration.spec.ts` (lines 26-37): maps `levelId` to `id` and drops authoring-only fields (`title`, `order`, `parTime`). This mirrors the private `toMinigameLevel()` in `minigame-play.ts`.

**Test structure:**
1. A `describe.each` block iterates all 21 levels for the 4 data validation tests (form spec, test cases, available elements, time limit) plus the `initialize()` smoke test
2. A separate test for level 1 evaluation with correct form element placement
3. A separate test for engine reset and reload with a different level

## Test Plan

### Tests 1-5: All 21 levels — engine initialization and signal validation

Use `describe.each` to iterate all 21 levels. Each level gets its own test block. Exact syntax:

```typescript
describe.each(TERMINAL_HACK_LEVELS)('level $levelId', (levelDef) => {
  let engine: TerminalHackEngine;

  beforeEach(() => {
    engine = new TerminalHackEngine();
    engine.initialize(toMinigameLevel(levelDef));
  });

  it('should load without errors', () => {
    expect(engine.status()).toBe(MinigameStatus.Loading);
    expect(engine.currentLevel()).toBe(levelDef.levelId);
  });

  it('should have at least 1 form element in engine signals', () => {
    expect(engine.targetFormSpec()!.elements.length).toBeGreaterThanOrEqual(1);
  });

  it('should have at least 1 test case in engine signals', () => {
    expect(engine.testCases().length).toBeGreaterThanOrEqual(1);
  });

  it('should have non-empty availableElements in engine signals', () => {
    expect(engine.availableElements().length).toBeGreaterThanOrEqual(1);
  });

  it('should have positive timeLimit in engine signals', () => {
    expect(engine.timeLimit()).toBeGreaterThan(0);
  });
});
```

**Key design decisions:**
- Assert ONLY via engine signals after `initialize()` — do NOT duplicate the raw data assertions already in `terminal-hack.data.spec.ts`. The unique value here is verifying the engine correctly loads and exposes level data.
- Each test creates a fresh engine in `beforeEach`, no cleanup needed since `initialize()` does not start timers.
- `describe.each` produces cleaner Vitest output than `it.each` with nested describes per level.

### Test 6: level 1 basic config produces expected evaluation results when correct form elements submitted

- Create engine with `TerminalHackFormEvaluationService`
- Initialize with level 1 (`th-basic-01`), start
- Place the correct element: `{ type: 'place-element', elementId: 'name', elementType: 'text', toolType: 'ngModel' }`
- Call `engine.runTestCases()`
- Assert: `result.allPassed === true`
- Assert: `result.passCount === 2` (level 1 has 2 test cases)
- Assert: `result.failCount === 0`
- Assert: `engine.status() === MinigameStatus.Won`

### Test 7: engine can be reset and loaded with a different level from the pack

- Create engine, initialize with level 1 (`TERMINAL_HACK_LEVELS[0]`), start
- Assert `engine.currentLevel()` equals `'th-basic-01'`
- Initialize with level 8 (`TERMINAL_HACK_LEVELS[7]`, `'th-intermediate-01'`) — this level uses reactive forms with `['FormControl']` tools, contrasting level 1's template-driven `['ngModel']`
- Assert `engine.status()` equals `MinigameStatus.Loading` (re-initialized)
- Assert `engine.currentLevel()` equals `'th-intermediate-01'`
- Assert `engine.targetFormSpec()!.formType` equals `'reactive'` (level 1 is `'template-driven'`, level 8 is `'reactive'` — proving the engine loaded the new level's data)
- Assert `engine.availableElements()` contains `'FormControl'` (not `'ngModel'`)
