# Plan: T-2026-016 — Define minigame lifecycle interfaces and types

## Task

Define the TypeScript interfaces and types for the minigame framework. These are the contracts that all 12 minigames will implement. All types exported from `src/app/core/minigame/minigame.types.ts` with JSDoc comments.

## Architecture

### Design Decisions

**1. Enums as `const enum` vs string literal unions vs regular enum.**
Decision: Use regular TypeScript `enum`. The ticket explicitly requests enums for `DifficultyTier` and `MinigameStatus`. Regular enums (not `const enum`) work correctly with Angular's template binding and are iterable at runtime, which is needed by downstream tickets (T-2026-046 TierBadge iterates tiers, T-2026-029 MinigameRegistryService filters by tier). String literal unions were considered but the ticket acceptance criteria specifically says "enum." Note: this diverges from the existing `Rank` type in `rank.constants.ts` which uses a string union -- this is intentional per ticket AC, establishing a new convention for enums where runtime iteration is needed.

**2. Generic `data` field on `MinigameLevel`.**
Decision: Use a generic type parameter `MinigameLevel<TData = unknown>`. Each minigame defines its own level data shape (Module Assembly has conveyor belt config, Wire Protocol has wiring puzzle config, etc.). The default `unknown` allows collections of mixed-type levels while preserving type safety when a specific minigame narrows the generic. This aligns with T-2026-017's `MinigameEngine<TLevelData>` and T-2026-019's `LevelDefinition<T>`.

**3. `MinigameConfig.difficultyTiers` field type.**
Decision: `DifficultyTier[]` -- an array of which tiers this minigame supports. All 12 minigames currently support all 4 tiers (Basic/Intermediate/Advanced/Boss per the TEMPLATE and all specs), but the array type allows future flexibility without breaking the interface.

**4. `MinigameConfig.angularTopic` type.**
Decision: `string` for now. A future ticket could introduce a `AngularTopic` string union from the curriculum data model (T-2026-038), but that creates a circular dependency if done now. Using `string` keeps this ticket self-contained.

**5. `MinigameId` branded type.**
Decision: Introduce a `MinigameId` string type alias for the 12 game IDs. This is a simple union type derived from the minigame specs: `'module-assembly' | 'wire-protocol' | ...`. This provides compile-time safety when passing game IDs between services (T-2026-029, T-2026-026, T-2026-042). The IDs use kebab-case to match URL slugs from the routing system (T-2026-012 already has `/minigames/:gameId`).

**6. File location.**
The ticket specifies `src/app/core/minigame/minigame.types.ts`. This creates a new `minigame/` directory under `core/`, which will grow to contain `minigame-engine.ts` (T-2026-017), `minigame-shell/` (T-2026-018), `score-calculation.service.ts` (T-2026-028), `minigame-registry.service.ts` (T-2026-029), and `hint.service.ts` (T-2026-044).

### Module Boundaries

- This file is pure types -- no Angular dependencies, no runtime imports, no services.
- Downstream consumers import individual types: `import { MinigameConfig, DifficultyTier } from '@app/core/minigame/minigame.types';` (or via barrel export once one is created).
- No barrel export for `core/minigame/` is created in this ticket -- that will come when the directory has more files (T-2026-017+). Note: the `core/` directory does not have a root `index.ts` barrel (unlike `shared/` which does per CLAUDE.md conventions). Downstream tickets T-2026-019 and T-2026-029 should import directly from `src/app/core/minigame/minigame.types` until a barrel is created.

### Relationship to Existing Code

- The pattern follows `src/app/core/state/rank.constants.ts` which defines types (`Rank`, `RankThreshold`) and constants (`RANK_THRESHOLDS`) in a pure TS file.
- JSDoc style follows the existing convention in `game-state.service.ts` (single-line `/** ... */` for brief docs, multi-line for complex explanations).

## Detailed Type Definitions

### `MinigameId` type alias

```typescript
export type MinigameId =
  | 'module-assembly'
  | 'wire-protocol'
  | 'flow-commander'
  | 'signal-corps'
  | 'corridor-runner'
  | 'terminal-hack'
  | 'power-grid'
  | 'data-relay'
  | 'reactor-core'
  | 'deep-space-radio'
  | 'system-certification'
  | 'blast-doors';
```

Derived from the 12 minigame spec files (`docs/minigames/01-12`). Kebab-case matches URL routing slugs.

### `DifficultyTier` enum

```typescript
export enum DifficultyTier {
  Basic = 'basic',
  Intermediate = 'intermediate',
  Advanced = 'advanced',
  Boss = 'boss',
}
```

String-valued enum for readability in serialized data and URL params. All 12 minigames use these 4 tiers per the TEMPLATE.

### `MinigameStatus` enum

```typescript
export enum MinigameStatus {
  Loading = 'loading',
  Playing = 'playing',
  Paused = 'paused',
  Won = 'won',
  Lost = 'lost',
}
```

String-valued for consistency with `DifficultyTier`. Used by `MinigameState.status` and T-2026-017's lifecycle state machine. Note: The ticket AC lists `status (playing/paused/won/lost)` as 4 values on `MinigameState`, but separately lists `Loading` as a required enum member on `MinigameStatus`. The `Loading` state is included in the enum as it is needed by T-2026-017's lifecycle (the pre-game state before level data is ready).

### `MinigameConfig` interface

```typescript
export interface MinigameConfig {
  readonly id: MinigameId;
  readonly name: string;
  readonly description: string;
  readonly angularTopic: string;
  readonly totalLevels: number;
  readonly difficultyTiers: readonly DifficultyTier[];
}
```

All fields `readonly` -- config is immutable after registration. `difficultyTiers` uses `readonly` array. Note: `totalLevels` varies per minigame (18 for most, 21 for Terminal Hack and Reactor Core).

### `MinigameLevel<TData>` interface

```typescript
export interface MinigameLevel<TData = unknown> {
  readonly id: string;
  readonly gameId: MinigameId;
  readonly tier: DifficultyTier;
  readonly conceptIntroduced: string;
  readonly description: string;
  readonly data: TData;
}
```

Generic `TData` defaults to `unknown`. Each minigame will define its own data shape (e.g., `ModuleAssemblyLevelData` with belt speed, parts, blueprint config).

### `MinigameState` interface

```typescript
export interface MinigameState {
  readonly currentLevel: string | null;
  readonly score: number;
  readonly lives: number;
  /** Countdown time remaining in seconds. 0 indicates time has expired. */
  readonly timeRemaining: number;
  readonly status: MinigameStatus;
}
```

Represents the runtime state of an active minigame session. `currentLevel` is a level ID string (or `null` before a level is loaded -- see updated interface below). `timeRemaining` is countdown time remaining in seconds (0 indicates expired). T-2026-017 will manage this via signals.

### `MinigameResult` interface

```typescript
export interface MinigameResult {
  readonly gameId: MinigameId;
  readonly levelId: string;
  readonly score: number;
  readonly perfect: boolean;
  readonly timeElapsed: number;
  readonly xpEarned: number;
}
```

Produced when a level is completed (won). Consumed by T-2026-020 (LevelProgressionService), T-2026-021 (XpService), and T-2026-022 (MasteryService). `gameId` included so downstream consumers can associate the result with a specific minigame without a secondary lookup from the level ID.

## Test Plan

The spec file (`minigame.types.spec.ts`) will verify:

1. **Enum value tests** (runtime checks):
   - `DifficultyTier.Basic` equals `'basic'`
   - `DifficultyTier.Intermediate` equals `'intermediate'`
   - `DifficultyTier.Advanced` equals `'advanced'`
   - `DifficultyTier.Boss` equals `'boss'`
   - `DifficultyTier` has exactly 4 members
   - `MinigameStatus.Loading` equals `'loading'`
   - `MinigameStatus.Playing` equals `'playing'`
   - `MinigameStatus.Paused` equals `'paused'`
   - `MinigameStatus.Won` equals `'won'`
   - `MinigameStatus.Lost` equals `'lost'`
   - `MinigameStatus` has exactly 5 members

2. **Interface shape compliance tests** (compile-time + runtime):
   - Create a valid `MinigameConfig` object literal and verify all properties are accessible
   - Create a valid `MinigameLevel` with default `unknown` data type
   - Create a typed `MinigameLevel<{ beltSpeed: number }>` and verify `data.beltSpeed` is accessible
   - Create a valid `MinigameState` and verify all properties
   - Create a valid `MinigameResult` and verify all properties

3. **MinigameId type tests** (compile-time via `tsc`, not runtime Jasmine assertions):
   - Assign each of the 12 valid game IDs to a `MinigameId` variable (verifies the union is complete). These are compile-time checks -- they pass the build step if correct, and fail `tsc` if an ID is misspelled or missing from the union. They will not appear as failing `it()` blocks during TDD Red step.
   - Use a `const ALL_GAME_IDS: MinigameId[]` array of all 12 IDs and verify its length equals 12 at runtime

4. **Enum member count tests** (runtime):
   - Count keys in `DifficultyTier` enum (filtering out reverse mappings -- not needed since string enums don't have them, but include the explicit count check)
   - Count keys in `MinigameStatus` enum
