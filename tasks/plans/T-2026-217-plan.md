RefresherChallengeService (T-2026-047) provides the logic, DegradationAlertComponent (T-2026-161) has a "Practice Now" button, but there is no page/route where the refresher challenge is actually played. This ticket creates the page that loads refresher challenge content and allows the player to complete it.

## Architecture

### Approach: State-machine page component with async loading

The RefresherChallengePage is a state-machine-driven component similar to MinigamePlayPage but much simpler -- it does not use minigame engines or MinigameShell. Instead, it follows a linear flow:

```
loading -> error | not-degrading | playing -> completed
```

**Why this approach:** Refresher challenges are described in progression.md as "quick refreshers: 3-5 questions." They use micro-levels from the existing level pack but do not require the full minigame engine machinery (no score, no lives, no timer, no pause). The page simply presents a sequence of level IDs that the player must complete, then records practice. This keeps the implementation lean and within the <200 LOC budget.

**Key design decisions:**

1. **No minigame engine integration in this ticket.** The page displays the micro-level IDs as a checklist of challenges. Actual minigame engine embedding (playing each micro-level inline) is a separate concern -- this ticket delivers the page shell, data loading, progress tracking, and mastery restoration. The micro-level display shows the level IDs with a "Complete" button per level. This is a pragmatic decision: wiring inline minigame engines into a non-MinigamePlayPage context would exceed 200 LOC and create tight coupling. A follow-up ticket can embed actual gameplay.

2. **State managed via signals.** The component uses Angular signals for viewState, challenge data, and current progress index. No RxJS subscriptions beyond the initial `toSignal` for route params.

3. **Existing shared components reused.** `MasteryStarsComponent` for before/after star display, `LoadingSpinnerComponent` for loading state, `ErrorStateComponent` for error state.

4. **MinigameRegistryService for topic name lookup.** The registry provides `getConfig(topicId).name` and `getConfig(topicId).angularTopic` for display.

### Data Flow

```
Route param (topicId)
  -> RefresherChallengeService.generateRefresher(topicId)
  -> RefresherChallenge { topicId, questions, gameId, microLevelIds, restoredStars }
  -> Player clicks through micro-levels (simplified: marks each complete)
  -> RefresherChallengeService.completeRefresher(topicId)
  -> SpacedRepetitionService.getEffectiveMastery(topicId) for before/after display
```

### View States

| State          | Condition                                                  | Display                                   |
| -------------- | ---------------------------------------------------------- | ----------------------------------------- |
| loading        | Challenge generation in progress                           | Loading spinner                           |
| error          | generateRefresher failed or returned null unexpectedly     | Error state with retry                    |
| not-degrading  | Topic is not currently degrading (no refresher needed)     | "All caught up!" message + back link      |
| playing        | Challenge loaded, player completing micro-levels           | Level checklist with progress             |
| completed      | All micro-levels marked complete, mastery restored         | Mastery restoration result + back link    |
