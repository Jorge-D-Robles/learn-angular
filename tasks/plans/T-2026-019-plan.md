## Task

Define the level data schema that all minigame levels use. Each minigame has game-specific level data, but the wrapper structure is shared. Types exported from `src/app/core/levels/level.types.ts`. Unit tests verify type compatibility via compile-time checks in a test file.

## Architecture

### Why a separate `levels/` module?

The existing `minigame/minigame.types.ts` already has a `MinigameLevel<TData>` interface, but it is intentionally a lighter runtime type for the minigame engine (no `order`, `title`, or `parTime`). The ticket defines `LevelDefinition<T>` as the **content authoring schema** -- the full data model used to define level packs, drive level selection UI, and feed into the progression system. These are different concerns:

- `MinigameLevel<T>` -- runtime engine type (what the engine needs to play a level)
- `LevelDefinition<T>` -- content/data type (what a level author defines)

Placing level content types under `src/app/core/levels/` keeps the separation clean and matches the directory pattern already established (`core/state/`, `core/minigame/`). Downstream tickets (`T-2026-020` LevelProgressionService, `T-2026-030` LevelLoaderService) also live under `src/app/core/levels/`.

### Type relationships

- `LevelDefinition<T>` imports `MinigameId` and `DifficultyTier` from `../minigame/minigame.types`
- `LevelPack` wraps an array of `LevelDefinition<unknown>` (heterogeneous game data)
- `LevelTierConfig` pairs a `DifficultyTier` with its XP reward and unlock requirement
- XP constants are `as const` objects, not an enum, following the pattern in `rank.constants.ts`

### Constants design

The XP reward values per tier and the perfect score multiplier are defined as named constants. Using a `Record<DifficultyTier, number>` map for tier XP rewards provides type-safe exhaustiveness -- if a new tier were ever added, TypeScript would flag the missing entry.

## Detailed Implementation

### `src/app/core/levels/level.types.ts`

```typescript
import { DifficultyTier } from '../minigame/minigame.types';
import type { MinigameId } from '../minigame/minigame.types';
```

**Interfaces:**

1. `LevelDefinition<T = unknown>` -- all fields `readonly`:
   - `levelId: string` -- unique level ID (e.g., `"ma-basic-01"`). Note: named `levelId` (not `id`) to distinguish from `MinigameLevel.id` in the engine types. Maps to `MinigameLevel.id` when passing to the engine.
   - `gameId: MinigameId` -- which minigame this level belongs to
   - `tier: DifficultyTier` -- difficulty tier
   - `order: number` -- sort order within the tier (1-based)
   - `title: string` -- human-readable level title (e.g., `"Minimal Component"`)
   - `conceptIntroduced: string` -- Angular concept practiced
   - `description: string` -- level objective description
   - `parTime?: number` -- optional par time in seconds for speed run mode. JSDoc: "Par time in seconds. Used for speed run scoring."
   - `data: T` -- game-specific level configuration data

2. `LevelPack` -- all fields `readonly`:
   - `gameId: MinigameId` -- which minigame this pack belongs to
   - `levels: readonly LevelDefinition<unknown>[]` -- all levels for this game

3. `LevelTierConfig` -- all fields `readonly`:
   - `tier: DifficultyTier` -- which tier
   - `xpReward: number` -- base XP awarded for completing a level of this tier
   - `unlockRequirement: string` -- human-readable unlock description (e.g., `"Complete all Basic levels"`)

**Constants:**

4. `TIER_XP_REWARDS: Readonly<Record<DifficultyTier, number>>`:
   - `[DifficultyTier.Basic]: 15`
   - `[DifficultyTier.Intermediate]: 20`
   - `[DifficultyTier.Advanced]: 30`
   - `[DifficultyTier.Boss]: 150`

5. `PERFECT_SCORE_MULTIPLIER = 2 as const`

6. `LEVEL_TIER_CONFIGS: readonly LevelTierConfig[]` -- array of all 4 tier configurations with unlock requirements:
   - Basic: `"Available from start"`
   - Intermediate: `"Complete all Basic levels"`
   - Advanced: `"Complete all Intermediate levels"`
   - Boss: `"Complete all Advanced levels"`

### `src/app/core/levels/level.types.spec.ts`

Follow the exact testing pattern from `minigame.types.spec.ts`:

**Compile-time type checks (top of file):**
- Create a `LevelDefinition` with default `unknown` data -- verifies the interface compiles
- Create a typed `LevelDefinition<{ beltSpeed: number }>` -- verifies generic works
- Create a `LevelDefinition` with `parTime` omitted -- verifies optionality
- Create a `LevelDefinition` with `parTime` included -- verifies optional field acceptance
- Create a `LevelPack` with heterogeneous level data types -- verifies `unknown[]` works
- Create a `LevelTierConfig` -- verifies the interface compiles
- Suppress unused variable warnings with `void [...]`

**Runtime test suites:**

1. `describe('LevelDefinition')`:
   - `it('should accept a valid level definition with all fields')` -- construct object, assert each field
   - `it('should accept a level definition without parTime')` -- construct without optional, verify undefined
   - `it('should accept typed game-specific data')` -- construct with `LevelDefinition<{ beltSpeed: number; partCount: number }>`, verify `data.beltSpeed` and `data.partCount` are accessible
   - `it('should accept different game data types')` -- construct a Wire Protocol level with `LevelDefinition<{ connections: { source: string; target: string }[] }>`, verifying different minigame data shapes work

2. `describe('LevelPack')`:
   - `it('should accept a valid level pack')` -- construct with `gameId` and an array of 2+ levels, assert `pack.gameId` and `pack.levels.length`
   - `it('should accept an empty levels array')` -- verify `levels: []` is valid

3. `describe('LevelTierConfig')`:
   - `it('should accept a valid tier config')` -- construct and assert each field

4. `describe('TIER_XP_REWARDS')`:
   - `it('should have Basic tier at 15 XP')` -- `expect(TIER_XP_REWARDS[DifficultyTier.Basic]).toBe(15)`
   - `it('should have Intermediate tier at 20 XP')` -- `expect(TIER_XP_REWARDS[DifficultyTier.Intermediate]).toBe(20)`
   - `it('should have Advanced tier at 30 XP')` -- `expect(TIER_XP_REWARDS[DifficultyTier.Advanced]).toBe(30)`
   - `it('should have Boss tier at 150 XP')` -- `expect(TIER_XP_REWARDS[DifficultyTier.Boss]).toBe(150)`
   - `it('should have exactly 4 entries')` -- `expect(Object.keys(TIER_XP_REWARDS).length).toBe(4)`

5. `describe('PERFECT_SCORE_MULTIPLIER')`:
   - `it('should be 2')` -- `expect(PERFECT_SCORE_MULTIPLIER).toBe(2)`

6. `describe('LEVEL_TIER_CONFIGS')`:
   - `it('should have exactly 4 tier configs')` -- `expect(LEVEL_TIER_CONFIGS.length).toBe(4)`
   - `it('should have XP rewards matching TIER_XP_REWARDS')` -- verify each config's `xpReward` matches the corresponding entry in `TIER_XP_REWARDS`
   - `it('should have correct unlock requirements for all tiers')` -- assert each config's `unlockRequirement` string: Basic="Available from start", Intermediate="Complete all Basic levels", Advanced="Complete all Intermediate levels", Boss="Complete all Advanced levels"
   - `it('should be ordered Basic, Intermediate, Advanced, Boss')` -- assert tier order matches progression
