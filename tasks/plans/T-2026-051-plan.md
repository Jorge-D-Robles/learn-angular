# Plan: T-2026-051 — Create AudioService for sound effect management

## Task

Multiple minigame specs reference sound effects and audio feedback (correct/incorrect sounds, completion jingles). The SettingsService already tracks `soundEnabled` preference. This service provides a centralized API for playing sound effects that respects the sound setting.

Acceptance criteria:
- `AudioService` at `src/app/core/audio/audio.service.ts`
- `play(soundId)`: plays a named sound effect if sound is enabled
- `SoundEffect` enum: correct, incorrect, complete, fail, levelUp, rankUp, hint, click, tick
- Respects `SettingsService.settings().soundEnabled` -- no-op when disabled
- Preloads sound files on initialization (lazy: only when first needed)
- Volume control (0-1 range)
- Does not block UI thread (uses Web Audio API or HTMLAudioElement)
- Unit tests for: play when enabled, no-op when disabled, volume control

## Architecture

### Approach: HTMLAudioElement with lazy-loaded Audio pool

**Why HTMLAudioElement over Web Audio API:**
- Web Audio API (`AudioContext`) requires user gesture to start on most browsers, adding complexity for a service that should "just work"
- HTMLAudioElement is simpler, well-supported, and sufficient for short sound effects
- HTMLAudioElement `.play()` returns a Promise (non-blocking by design)
- For a gamified learning app, we do not need low-latency mixing or audio processing -- just fire-and-forget playback

**Lazy preloading strategy:**
- Sound files are NOT loaded at service construction time
- On the first call to `play()` (or an explicit `preload()` call), the service creates `HTMLAudioElement` instances for all sounds and begins loading them via the browser's built-in `preload="auto"` mechanism
- A `Map<SoundEffect, HTMLAudioElement>` stores the cached elements
- Subsequent `play()` calls clone the cached element to allow overlapping playback (`.cloneNode(true) as HTMLAudioElement` — cast required since `cloneNode()` returns `Node`)

**Volume control:**
- A `_volume` private signal (0-1, default 0.5), exposed as `volume = this._volume.asReadonly()` (consistent with GameStateService pattern)
- `setVolume(v: number)` clamps to [0, 1] and updates the private signal — this is the only mutation point
- Volume is applied to each audio element at play time

**Sound file paths:**
- Sound files will live at `public/audio/{soundId}.mp3` (or `.ogg`, `.wav`)
- Since no actual sound files exist yet, the service maps `SoundEffect` enum values to file paths via a constant `SOUND_PATHS` record
- File format is `.mp3` (most universally supported)
- The path prefix is configurable via a private constant `AUDIO_BASE_PATH = 'audio/'`

**Integration with SettingsService:**
- Injects `SettingsService` and reads `settings().soundEnabled` on every `play()` call
- When disabled, `play()` is an immediate no-op (no audio elements created, no errors)
- The service does NOT subscribe reactively to settings -- it checks at play time, which is simpler and avoids unnecessary wiring

### Module boundaries

- `AudioService` is `providedIn: 'root'` (same pattern as all core services)
- `SoundEffect` enum is exported from the same file (co-located, since it is the service's public API type)
- Barrel export (`index.ts`) is NOT created here -- that is T-2026-165's responsibility
- The core root barrel (`src/app/core/index.ts`) is NOT modified here -- also T-2026-165

### Edge cases

- **No audio file found:** `play()` catches errors from `HTMLAudioElement.play()` and logs a warning. Does not throw.
- **Rapid repeated plays:** Each `play()` call clones the cached audio element, so overlapping sounds work naturally.
- **SSR / no window:** Guard `typeof window !== 'undefined'` before creating audio elements. In non-browser contexts, `play()` is a no-op.
- **Volume out of range:** `setVolume()` clamps to [0, 1].
- **play() called before preload completes:** The browser handles this gracefully -- `HTMLAudioElement.play()` will either play from whatever has loaded or wait. We fire-and-forget.

Override `window.Audio` via `vi.stubGlobal('Audio', MockAudio)`. For test 10, mock `play()` to reject. ## Implementation Order

1. **Define `SoundEffect` enum and `SOUND_PATHS` constant** -- The enum lists all 9 sound effect IDs. The `SOUND_PATHS` record maps each enum value to a relative file path (e.g., `audio/correct.mp3`). This is pure data with no dependencies.

2. **Implement `AudioService` class** -- `@Injectable({ providedIn: 'root' })` service with:
   - `inject(SettingsService)` for `soundEnabled` check
   - Private `_cache: Map<SoundEffect, HTMLAudioElement>` for preloaded elements
   - Private `_preloaded = false` flag
   - `volume` signal (default 0.5) and `setVolume(v: number)` method
   - `preload()` method: guards with `if (this._preloaded) return;` for idempotency, sets `_preloaded = true` immediately, then iterates `SOUND_PATHS`, creates `new Audio(path)` for each, stores in cache
   - `play(soundId: SoundEffect)` method: checks `soundEnabled`, lazy-preloads if needed, clones cached element, sets volume, calls `.play()`, catches errors

3. **Write unit tests** -- Following the test plan above, using a mock `Audio` constructor and spying on `SettingsService.settings`.
