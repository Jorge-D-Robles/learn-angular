# Plan: T-2026-510 — Create TerminalHackTestRunnerComponent for test case execution panel

## Task

Terminal Hack spec describes: "Test inputs are provided -- the form must handle them correctly" and "Test button -- run predefined test inputs against the form." The test runner panel displays test cases, their execution status, and pass/fail indicators. Extracting this into a sub-component follows the established pattern (like TerminalHackLivePreviewComponent and TerminalHackCodePanelComponent).

Acceptance criteria:
- `TerminalHackTestRunnerComponent` at `src/app/features/minigames/terminal-hack/test-runner/test-runner.ts`
- Input: `testCases` (FormTestCase[]) -- the level's predefined test cases
- Input: `testResults` (TestCaseResult[] | null) -- results after running tests (null before first run)
- Input: `isRunning` (boolean) -- whether tests are currently executing
- Displays each test case with description and expected outcome
- After run: pass/fail icons per test (green checkmark / red X)
- Pass rate counter: "5/7 tests passed"
- Test input animation: when running, inputs animate into form fields (per spec)
- Output: `runTestsRequested` event
- Exported from terminal-hack barrel
- Unit tests for: test case listing, pass/fail rendering, pass rate display, running state, run request event

## Architecture

### Module boundaries

This is a **presentational (dumb) sub-component** that receives data via inputs and emits events via outputs. It has zero knowledge of the engine, services, or routing. The parent `TerminalHackComponent` will wire it up (in T-2026-512, not this ticket).

### Data flow

```
Parent (TerminalHackComponent)
  |-- [testCases]="..." --------> TerminalHackTestRunnerComponent
  |-- [testResults]="..." ------> (displays pass/fail per test)
  |-- [isRunning]="..." --------> (shows running animation)
  <-- (runTestsRequested) -------- (user clicks "Run Tests" button)
```

### Design decisions

1. **Follow the sibling component pattern exactly.** The live-preview and code-panel sub-components both use `input()` / `input.required()` / `output()` from `@angular/core`, inline templates, and BEM-style SCSS. This component will do the same.

2. **Selector: `app-terminal-hack-test-runner`.** All Terminal Hack components use the `app-` prefix per project convention.

3. **Pass rate is computed from inputs.** The component computes `passCount` and `totalCount` from the `testResults` input rather than receiving them as separate inputs, keeping the API surface small. The parent already has a `testRunResult` signal with `passCount`/`failCount`, but the sub-component re-derives from the array because it only receives `TestCaseResult[]`, not the aggregate `TestRunResult`.

4. **Animation approach.** The spec says "test execution: inputs animate into form fields." For the test runner panel specifically, this translates to a CSS class-based animation on individual test case rows that triggers when `isRunning` transitions from false to true. A `.test-runner__case--animating` class will use a staggered `animation-delay` per row, creating a sequential "checking each test" visual. When `prefers-reduced-motion` is set, the animation is disabled. The animation is purely CSS, not runtime JS, keeping the component simple.

5. **Test case display format.** Each test case row shows:
   - Description text (from `FormTestCase.description`)
   - Expected outcome badge ("Valid" / "Invalid" from `expectedValid`)
   - After run: pass/fail icon (checkmark or X) with color coding
   - During run: a pulsing/scanning animation

6. **Pass rate counter format.** Displayed as "N/M tests passed" (e.g., "5/7 tests passed") per the acceptance criteria. This appears only when `testResults` is non-null.

7. **Types imported from existing files.** `FormTestCase` from `terminal-hack.types.ts` and `TestCaseResult` from `terminal-hack.engine.ts` are already defined and exported.

8. **Input declarations:**
   - `testCases = input.required<FormTestCase[]>()` — always provided by parent
   - `testResults = input<readonly TestCaseResult[] | null>(null)` — `readonly` to be forward-compatible with engine's `readonly TestCaseResult[]`; `null` before first run
   - `isRunning = input<boolean>(false)` — defaults to false
   - `runTestsRequested = output<void>()`

9. **Animation mechanism:** Apply `--animating` class while `isRunning()` is true via `[class.test-runner__case--animating]="isRunning()"`. Use CSS `animation` (not `transition`) with `animation-fill-mode: forwards` so each row fires its stagger once on class add. Use `[style.animation-delay]="i * 80 + 'ms'"` via the `@for` loop's `$index` to stagger rows.

10. **Test 16 clarification:** When `testCases: []` and `testResults: []`, the pass rate element IS visible and reads "0/0 tests passed" (since `testResults` is non-null). Zero case rows are rendered.

## Implementation Order

1. **Create `test-runner.ts`** -- The component file with inline template. Import `FormTestCase` from `terminal-hack.types` and `TestCaseResult` from `terminal-hack.engine`. Define inputs (`testCases`, `testResults`, `isRunning`), one output (`runTestsRequested`), and computed signals for `passCount`, `totalCount`, and `passRateText`.
